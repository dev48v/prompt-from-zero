🌊 Streaming — stop waiting for the whole answer. Stream tokens over SSE and render them the instant they arrive, so the user watches the reply type out in real time. At the same total time, streaming feels ~5–10× faster because what people actually feel is time-to-first-token, not time-to-complete — plus you can cancel mid-stream and keep the partial output.

Run the same response two ways — streamed vs buffered — and watch which one feels fast.

A language model produces its answer token by token. You can either buffer — wait for the entire response, then show it all at once — or stream — push each token to the client the moment it's generated and render it immediately. Both finish at the exact same total time. But the streamed version starts showing words after a few hundred milliseconds (its time-to-first-token, TTFT), while the buffered version shows a spinner for the whole duration and then dumps a wall of text. Since the human brain measures the wait as "how long until something happened", streaming can feel 5–10× faster at identical throughput. 🔑 Streaming = ship tokens as they're generated, not after. The transport is SSE / chunked HTTP: one long response, many small frames. Total generation time doesn't change — but TTFT collapses from "the whole answer" to "a few hundred ms", and because people feel the first-content wait, the app reads as far faster. You render incrementally (append each token) and, for structured output, render only the fields that have fully arrived — never JSON.parse a half-arrived buffer. Streaming also makes work cancellable: abort mid-stream and the partial output is still on screen and useful. Skip streaming when you need one validated object, for batch jobs, tool-only turns, or responses so short buffering is already instant.

🔨 Build it yourself

    
Streaming is one delivery pattern layered onto a normal request: server flushes tokens over SSE → client reads the stream frame-by-frame → render each token → cancel with AbortController. The whole thing is small. Here it is end to end: the server side that emits Server-Sent Events, the wire format (and how providers differ), the client reader loop, incremental frame parsing, safe incremental rendering, and cancellation — plus when not to stream.

    
    
      
  1. 
        
    1
    Server: emit tokens as SSE, and flush
    
        
    Set the text/event-stream content type and write one small frame per token as the model produces it. The key detail is flushing — without it a proxy or framework buffer will hold the whole body and silently un-stream your stream. End with a sentinel so the client knows generation finished.
    
        
    // SERVER — emit tokens the moment they are generated (Server-Sent Events).
res.writeHead(200, {
  "Content-Type": "text/event-stream",  // the SSE content type
  "Cache-Control": "no-cache",
  "Connection": "keep-alive",
});
for await (const token of model.generate(prompt)) {
  res.write("data: " + JSON.stringify({ token }) + "\n\n"); // one SSE frame
  res.flush?.();   // flush! or a proxy may buffer the whole response
}
res.write("data: [DONE]\n\n");   // sentinel: generation finished
res.end();
    
      
    2. 
      
  2. 
        
    2
    The wire format — and how providers frame it
    
        
    SSE is just a long HTTP response made of data: lines, each terminated by a blank line. Providers layer their own event shape on top, but the idea is identical: many small chunks instead of one big body. Know your provider's framing so your parser matches it.
    
        
    // THE WIRE FORMAT — Server-Sent Events over one long HTTP response.
// Each event is a "data:" line, terminated by a blank line:
//
//   data: {"token":"Stream"}
//   data: {"token":"ing"}
//   data: [DONE]
//
// Providers frame it differently, same underlying idea:
//   OpenAI / Vercel AI SDK  ->  data: {...} lines, ending with  data: [DONE]
//   Anthropic (Claude)      ->  typed events: message_start,
//                               content_block_delta (text_delta),
//                               message_delta, message_stop  (no [DONE])
// Match your reader to the provider's event shape.
    
      
    3. 
      
  3. 
        
    3
    Client: open the stream, read chunks
    
        
    On the client, fetch() the endpoint and read res.body as a ReadableStream. Each reader.read() yields whatever bytes have arrived — not aligned to token or line boundaries — so you decode into a running buffer and pull complete frames out of it. Pass an AbortController signal now; you'll use it to cancel in step 6.
    
        
    // CLIENT — open the stream and read chunks as they arrive.
const res = await fetch("/api/chat", {
  method: "POST",
  body: JSON.stringify({ prompt }),
  signal: controller.signal,          // for cancellation (step 6)
});
const reader  = res.body.getReader(); // a ReadableStream reader
const decoder = new TextDecoder();
let buffer = "";
while (true) {
  const { value, done } = await reader.read();
  if (done) break;                    // stream closed by server
  buffer += decoder.decode(value, { stream: true });
  // ...pull complete SSE frames out of `buffer` (step 4)
}
    
      
    4. 
      
  4. 
        
    4
    Parse SSE frames incrementally
    
        
    Chunks split mid-line, so you can't parse a chunk directly. Instead, scan the buffer for the frame delimiter (a blank line, "\n\n"), process each complete frame, and leave the unfinished remainder in the buffer for the next read. This is the one bug everyone hits first: parsing a half-arrived line.
    
        
    // PARSE SSE FRAMES INCREMENTALLY — chunks split mid-line, so buffer them.
let idx;
while ((idx = buffer.indexOf("\n\n")) !== -1) {
  const frame = buffer.slice(0, idx);
  buffer      = buffer.slice(idx + 2);   // keep the unfinished remainder
  const line  = frame.replace(/^data: /, "");
  if (line === "[DONE]") return finish();
  const { token } = JSON.parse(line);    // one line == one COMPLETE JSON
  onToken(token);                        // append to the UI (step 5)
}
// a partial trailing frame stays in `buffer` until the next read()
    
      
    5. 
      
  5. 
        
    5
    Render incrementally — safely for structured data
    
        
    For prose, rendering is trivial: append each token. For structured output (JSON), the accumulated buffer is invalid JSON for most of the stream — don't JSON.parse it every tick or it throws until the final byte. Render only the fields that have fully arrived (or use a tolerant streaming parser), so the half-arrived field simply waits its turn instead of crashing the UI.
    
        
    // RENDER INCREMENTALLY — append tokens; keep partial output safe.
function onToken(t) {
  answerEl.textContent += t;      // prose: user watches it type out
}

// Streaming STRUCTURED data? The buffer is invalid JSON mid-stream.
// Do NOT JSON.parse the partial each tick — it throws until the last byte.
// Render only the fields that have fully arrived:
function renderPartial(buf) {
  for (const [k, v] of completedFields(buf)) {  // e.g. "key":"value" pairs
    setField(k, v);               // a half-arrived field waits its turn
  }
}
// (or feed the buffer to a tolerant / streaming JSON parser)
    
      
    6. 
      
  6. 
        
    6
    Cancel mid-stream — and when NOT to stream
    
        
    Because you hold the connection open, you can abort it. controller.abort() tears down the fetch; the server sees the client drop and stops generating (so you stop paying for tokens). Whatever streamed already stays on screen and is still useful — the opposite of a buffered request, where aborting leaves you with nothing. Streaming isn't always right, though. Next up you'll switch topics to system prompts: the persona-and-rules turn that shapes everything the model says.
    
        
    // CANCEL MID-STREAM — stop the model, stop paying for tokens.
const controller = new AbortController();
sendBtn.onclick = () => startStream(controller);  // passes signal to fetch
stopBtn.onclick = () => controller.abort();        // tears down the stream
// abort() closes the connection; the server sees the drop and halts.
// Whatever already streamed stays on screen (usable) — unlike a buffered
// request, where aborting leaves you with nothing.

// WHEN NOT TO STREAM:
//  - you need ONE validated object (parse the whole thing, then act)
//  - non-interactive / batch jobs (no human waiting on TTFT)
//  - tool-call-only turns the user never reads
//  - tiny responses where buffering is already instant
