🔁 Rephrase-and-Respond (RaR) — make the model restate your question before it answers, so it locks onto what you actually meant

A human silently disambiguates your question. The model doesn't — unless you make it.

Human questions are full of small ambiguities — an "even month", a bare date like 02/03, a temperature with no unit. You resolve them without noticing; a language model just picks a reading — often not the one you meant — and answers confidently on the wrong interpretation. Rephrase-and-Respond (RaR) fixes this cheaply: before answering, the model restates and expands your question in its own words, which forces it to commit to one interpretation on the record — and that act of restating usually surfaces the ambiguity and lands on the intended reading. 🔑 The rephrase is the whole trick. Nothing about the model's knowledge changed between the two columns — it knew Tom Cruise's birthday, both date formats, both temperature scales all along. What changed is that RaR made it say out loud which question it was answering before answering. That single restatement forces the model to resolve the underspecified part explicitly, so instead of quietly betting on one reading it either picks the intended one or flags that the question is ambiguous. Same model, same prompt cost of a sentence or two — dramatically less silent misreading.

🔨 Build it yourself

    
RaR is almost pure prompt engineering — you add one instruction that tells the model to restate and expand the question before answering. Here is the one-step template, the two-step variant, how to feed the rephrasing into a Chain-of-Thought answer, and how to decide when it's worth the extra tokens.

    
    
      
  1. 
        
    1
    The one-step RaR prompt — rephrase, then answer, in one call
    
        
    The whole technique can be a single appended instruction. You ask the model to rewrite and expand the question in its own words first, then answer that rewrite. The rephrasing forces it to commit to an interpretation on the record before it produces an answer.
    
        
    const RAR_ONE_STEP = `${question}

Rephrase and expand the question in your own words, spelling out any
part that could be read more than one way and stating the reading you
will use. Then, on a new line, answer the question you rephrased.`;

const reply = await llm(RAR_ONE_STEP);
// reply:
//   Rephrased: You're asking whether the NUMBER of Tom Cruise's
//   birth month (Jan=1 … Dec=12) is even. He was born July 3, 1962;
//   July is month 7.
//   Answer: No — 7 is odd, so it is not an even-numbered month.
    
      
    2. 
      
  2. 
        
    2
    The two-step variant — one call to rephrase, one to answer
    
        
    Split the work across two calls. The first returns only the rephrased question; the second answers that clean, disambiguated version. It costs an extra round-trip but the rephrasing becomes an inspectable artifact — you can log it, show it to the user for confirmation, or route on it.
    
        
    // call 1 — rephrase only
const rephrased = await llm(
  `Rewrite and expand this question in your own words, making every
   ambiguous part explicit. Output ONLY the rewritten question:
   "${question}"`);

// call 2 — answer the rephrased question
const answer = await llm(rephrased);

return { rephrased, answer };   // both are inspectable
    
      
    3. 
      
  3. 
        
    3
    Trigger it only when the question is ambiguous
    
        
    RaR earns its extra tokens on underspecified questions and wastes them on crisp ones. A cheap heuristic (or a tiny classifier prompt) decides whether to rephrase first. When in doubt, rephrasing rarely hurts correctness — it only costs latency.
    
        
    function looksAmbiguous(q) {
  return /\b(even|odd) (month|day)\b/i.test(q)   // "even month"
      || /\b\d{1,2}\/\d{1,2}\b/.test(q)          // bare date 02/03
      || /\b\d+\s*(degrees|deg)\b/i.test(q)      // unit-less temp
      || /\b(it|this|that|they)\b/i.test(q);     // dangling pronoun
}
const answer = looksAmbiguous(q)
  ? await rar(q)          // rephrase first
  : await llm(q);         // answer directly
    
      
    4. 
      
  4. 
        
    4
    Combine RaR with Chain-of-Thought
    
        
    RaR and CoT fix different failure modes and stack cleanly: rephrase to clarify the question, then reason step-by-step toward the answer. Do the rephrase first so the reasoning chain is anchored to the correct interpretation from its first token.
    
        
    const RAR_COT = `${question}

Step 1 — Rephrase: restate the question in your own words, making
         every ambiguous part explicit and stating your interpretation.
Step 2 — Reason: think through it step by step for the rephrased version.
Step 3 — Answer: give the final answer.`;

const reply = await llm(RAR_COT);
// clarify the question (RaR) THEN reason about it (CoT)
    
      
    5. 
      
  5. 
        
    5
    Self-clarify vs. ask the user back
    
        
    RaR resolves ambiguity itself and states its assumption — no round-trip with the user, which keeps single-shot pipelines flowing. When a wrong guess is expensive, add a branch: if the rephrasing exposes readings that give genuinely different answers, surface them instead of silently choosing.
    
        
    const { rephrased, readings } = await rephraseAndDetect(q);
if (readings.length > 1 && answersDiffer(readings)) {
  // wrong guess is costly -> ask the user
  return `Did you mean: ${readings.map(r => r.label).join("  or  ")}?`;
}
// otherwise answer under the stated interpretation
return await llm(rephrased);
    
      
    6. 
      
  6. 
        
    6
    Weigh the cost — and keep the rephrasing honest
    
        
    Every RaR answer carries a rephrasing you didn't ask for, so it costs extra tokens and latency, and in chat UIs it can read as padding — hide the restatement or keep it to one line. Guard against a subtler failure: a rephrase that quietly changes the question. Validate that the restatement preserves the ask before trusting the answer.
    
        
    const { rephrased, answer } = await rar(q);
// cheap sanity check: the rephrase must keep the key terms
const kept = keyTerms(q).every(t => rephrased.toLowerCase().includes(t));
if (!kept) return await llm(q);   // rephrase drifted -> fall back
// optionally strip the restatement from what the user sees
return stripRephrase(answer);
