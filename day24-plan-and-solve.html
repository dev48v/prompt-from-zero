🗂️ Plan-and-Solve — plan the subtasks, then execute

Same problem, three prompts — only one keeps every step.

Multi-step word problems break plain LLMs in two ways: they skip a step (missing-step error) or they fumble the arithmetic (calculation error). Watch three prompting styles attack the same problem. (a) Plain just blurts an answer and quietly drops a step. (b) Zero-shot CoT — "let's think step by step" — reasons more but can still slip. (c) Plan-and-Solve first writes an explicit PLAN of numbered subtasks, then executes each one while tracking every variable — so no step goes missing and the intermediate numbers are visible. Press ▶ and watch the plan path land the right number while the naive path lands the classic wrong one. Real deterministic arithmetic over a scripted problem — no live model.

🔨 Build it yourself

    
Plan-and-Solve is not a new architecture — it's a trigger phrase you paste after the question, in place of "let's think step by step". You steer one zero-shot call to (1) understand the problem and devise a plan of subtasks, then (2) carry the plan out in order. PS+ adds two more instructions that kill the two dominant error types: extract the relevant variables/intermediates, and calculate carefully.

    
    
      
  1. 
        
    1
    Baseline: the zero-shot CoT trigger
    
        
    Start from what you already know. Appending Let's think step by step. to a question turns a plain answer into a reasoning chain and beats plain prompting on math word problems — but the chain often drops a step or miscomputes. Plan-and-Solve is a drop-in upgrade to this exact slot.
    
        
    const question = "A shop had 120 apples. It sold 40% in the morning\nand 15 in the afternoon. How many are left?";

// Zero-shot CoT — the Day-2 trigger:
const cotPrompt = question + "\nLet's think step by step.";
    
      
    2. 
      
  2. 
        
    2
    The Plan-and-Solve (PS) trigger
    
        
    Swap the trigger. Ask the model to first understand the problem and devise a plan, then carry out the plan. Forcing an explicit plan first makes the model enumerate the subtasks — so it is far less likely to silently skip one.
    
        
    const PS = "Let's first understand the problem and devise a plan\n"
  + "to solve it. Then, let's carry out the plan and solve the\n"
  + "problem step by step.";

const psPrompt = question + "\n" + PS;   // one zero-shot call
    
      
    3. 
      
  3. 
        
    3
    PS+ — extract variables & compute intermediates
    
        
    The two failure modes are missing steps and bad arithmetic. PS+ patches both by adding explicit instructions: pull out the relevant variables and their numbers, and compute intermediate results as you go. Naming each intermediate keeps the model from losing the thread.
    
        
    const PS_PLUS = "Let's first understand the problem, extract relevant\n"
  + "variables and their corresponding numerals, and devise a\n"
  + "plan. Then, let's carry out the plan, calculate intermediate\n"
  + "results (pay attention to calculation and commonsense),\n"
  + "and solve the problem step by step.";

const psPlusPrompt = question + "\n" + PS_PLUS;
    
      
    4. 
      
  4. 
        
    4
    Add "calculate carefully" to fight arithmetic slips
    
        
    The parenthetical (pay attention to calculation and commonsense) is doing real work: it nudges the model to double-check each operation instead of rushing the last line. This is the cheapest available guard against the calculation-error failure mode.
    
        
    // The clause that reduces calculation errors:
"...calculate intermediate results "
+ "(pay attention to calculation and commonsense)..."
// Keep intermediates named so a wrong number is easy to spot.
    
      
    5. 
      
  5. 
        
    5
    Parse the plan, then verify each subtask ran
    
        
    Because PS makes the plan explicit, you can machine-check it: split the plan into numbered subtasks and confirm the solution addresses each one. A subtask with no matching execution line is your missing-step alarm — catch it and re-ask.
    
        
    const out  = await llm(psPlusPrompt);
const plan = out.match(/(?:Step|Plan)\s*\d+[:.].*/gi) || [];
const solved = plan.filter(p => out.includes(resultFor(p)));
if (solved.length < plan.length)
  retry("A planned subtask was never executed.");
    
      
    6. 
      
  6. 
        
    6
    Know the limits — and when to escalate
    
        
    PS is still one zero-shot pass: the model executes its own plan and can still err on very long or symbolic problems. When correctness must be higher, escalate — self-consistency (sample many PS chains, vote), least-to-most (solve subproblems in dependency order), or hand the arithmetic to a tool. PS is the cheap, no-example first line of defence.
    
        
    // Escalation ladder (cheap → expensive):
// 1. PS / PS+           (this page — 1 call, no examples)
// 2. self-consistency   (sample N PS chains, majority vote)
// 3. least-to-most      (explicit subproblem decomposition)
// 4. tool / code exec   (offload the actual arithmetic)
