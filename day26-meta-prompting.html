🎼 Meta-Prompting — use the model to write & refine your own prompts

Don't hand-craft the prompt — let the model write the prompt.

A weak prompt like "write about dogs" isn't wrong, it's underspecified: role, audience, format, length and constraints are all left blank, so the model fills each with its blandest default. Meta-prompting turns prompt-engineering itself into a task for the model. A reusable meta-prompt — a fixed template carrying a quality checklist — 1️⃣ CRITIQUES your rough ask against every dimension a strong prompt should specify, then 2️⃣ REWRITES it into a structured prompt (role · context · constraints · format · examples). Press ▶ and watch the meta-prompt diagnose the gaps and compose the strong prompt, then see the same model, same topic produce a bland blob from the weak prompt versus a specific, structured answer from the rewritten one. 🎛 Conductor / orchestrator variant. A meta-prompt doesn't have to rewrite a single prompt. A conductor prompt takes a big goal — "launch my product" — decomposes it into several specialised sub-prompts (a market-analyst prompt, a copywriter prompt, a checklist-builder prompt), dispatches each to a fresh model instance with its own persona, then merges the results. The conductor never does the object-level work itself; it writes and coordinates the prompts that do. Same idea, one level up — the orchestration pattern behind ReWOO (Day 23) and Graph-of-Thoughts (Day 21), but the units being coordinated are prompts the model wrote for itself.

The meta-prompt makes every silent default an explicit decision: it names the ten dimensions a strong prompt should specify, flags the ones your draft skipped, and rewrites them into a role/context/constraints/format/example block. The vague ask and the structured ask go to the same model — only the specification quality differs, and that difference is exactly what a meta-prompt automates. 🔨 Build it yourself

    
Meta-prompting is not a new architecture — it's one reusable prompt that upgrades other prompts. You encode what a good prompt looks like as a checklist, wrap it in critique-and-rewrite instructions, and feed it your rough ask. The template flavour below rewrites once per prompt for a single extra call; the optimizer flavour (APE/OPRO) and the conductor variant scale the same idea up. Here's the whole pipeline.

    
    
      
  1. 
        
    1
    Encode what a good prompt looks like
    
        
    Write down the dimensions a strong prompt should specify. This checklist is the backbone of the meta-prompt — it turns "make it better" into a concrete list of things to check. Not every task needs all ten, but naming each one forces a deliberate choice instead of a silent default.
    
        
    const CHECKLIST = [
  "ROLE — the persona the model should adopt",
  "AUDIENCE — who the output is for",
  "CONTEXT — the situation / where this lives",
  "TASK — the precise, unambiguous ask",
  "CONSTRAINTS — scope, bans, must-dos",
  "FORMAT — the shape of the output",
  "LENGTH — how long",
  "TONE — style and voice",
  "EXAMPLES — a pattern to imitate",
  "SUCCESS CRITERIA — what 'done' means"
];
    
      
    2. 
      
  2. 
        
    2
    Write the reusable meta-prompt
    
        
    Wrap the checklist in critique-then-rewrite instructions, with one slot for the user's rough ask. This is the whole reusable artifact — a fixed super-prompt you author once and point at any weak prompt. Keep it in your repo like a house style guide.
    
        
    const META_PROMPT = `
You are a prompt engineer. Improve the user's prompt.

STEP 1 — CRITIQUE: for each item in the checklist below,
say whether the prompt specifies it, and what is missing.
STEP 2 — REWRITE: output a new prompt that fills every
gap, as a labelled block (ROLE, AUDIENCE, CONTEXT, TASK,
CONSTRAINTS, FORMAT, LENGTH, TONE, EXAMPLE, SUCCESS).

Checklist:
${CHECKLIST.join("\n")}

User's prompt:
"""{{WEAK_PROMPT}}"""`;
    
      
    3. 
      
  3. 
        
    3
    CRITIQUE — diagnose the gaps first
    
        
    Run the meta-prompt and let the model list, explicitly, what the weak prompt leaves unstated. Asking for the diagnosis before the rewrite matters: it makes the model reason about specific defects, so the rewrite is grounded in named gaps rather than a vague urge to improve.
    
        
    function metaPrompt(weak) {
  return META_PROMPT.replace("{{WEAK_PROMPT}}", weak);
}
// CRITIQUE output, e.g. for "write about dogs":
//  ✗ no ROLE  ✗ no AUDIENCE  ✗ TASK vague
//  ✗ no FORMAT  ✗ no LENGTH  ✗ no SUCCESS CRITERIA
const critique = await llm(metaPrompt("write about dogs"));
    
      
    4. 
      
  4. 
        
    4
    REWRITE — emit the structured prompt
    
        
    The same call then rewrites the prompt, filling every gap it just found into a labelled block. This structured prompt — not the critique — is the artifact you actually run. It reads like a spec because it is one.
    
        
    // REWRITE output (the prompt you'll actually run):
const strongPrompt = `
ROLE: You are a veterinary science writer.
AUDIENCE: First-time dog owners, no prior knowledge.
CONTEXT: A blog series on responsible pet ownership.
TASK: Write a beginner's guide to choosing a first breed.
CONSTRAINTS: Exactly 5 breeds; note energy, size, grooming;
             no medical advice.
FORMAT: Markdown — one H2 per breed + a summary table.
LENGTH: ~600 words.  TONE: warm, practical, jargon-free.
EXAMPLE: "Labrador — high energy, large, low grooming."
SUCCESS: Reader can pick a breed matching their lifestyle.`;
    
      
    5. 
      
  5. 
        
    5
    Run the improved prompt (and optionally loop)
    
        
    Now send the rewritten prompt to the model to get the real answer. Because the meta-prompt is reusable, you spend just one extra call per prompt. You can even loop it — critique the rewrite and rewrite again — until it stops finding gaps, the same critique→revise idea as self-refine (Day 12) applied to the prompt itself.
    
        
    const strong = await metaPrompt(weak);   // 1 extra call
const answer = await llm(strong);         // run the good prompt

// optional refinement loop:
let p = weak;
for (let i = 0; i < 2; i++) p = await metaPrompt(p);
const best = await llm(p);
    
      
    6. 
      
  6. 
        
    6
    Scale it up — optimizer & conductor, and the limits
    
        
    The template flavour rewrites once with no data. To go further, the optimizer flavour (APE, OPRO) proposes many candidate prompts and iterates toward the best-scoring one against an eval set; the conductor flavour decomposes a goal into sub-prompts and orchestrates them. Both are still ordinary LLM calls, so review the output: a rewrite can invent constraints you never wanted, or just get longer without getting better, and a meta-prompt is only as good as its checklist.
    
        
    // Optimizer (needs a scored eval set):
let history = [];
for (let i = 0; i < N; i++) {
  const cand  = await llm(`Propose a better prompt.\n${history}`);
  const score = evaluate(cand, examples);
  history.push({ cand, score });     // OPRO/APE search
}
// Conductor: goal -> sub-prompts -> merge (Day 21/23).
// Always: review(strong) — did it invent constraints?
