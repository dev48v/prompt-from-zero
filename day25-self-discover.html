🧠 Self-Discover — the model composes its own reasoning structure

Don't reason the same way for every task — discover the structure first.

Plain chain-of-thought applies one fixed reasoning style to every problem: "let's think step by step". But different tasks need different reasoning. Self-Discover (Google DeepMind, 2024) gives the model a menu of ~39 atomic reasoning modules and runs three meta-steps: 1️⃣ SELECT the few modules relevant to this task, 2️⃣ ADAPT their wording to the specific problem, and 3️⃣ IMPLEMENT them into a task-specific reasoning structure (a small JSON plan). Only then does it SOLVE — by filling in that bespoke structure. Press ▶ and watch the structure get built for this task, then executed with real arithmetic — while plain CoT charges ahead with its one-size-fits-all chain and trips on the exact thing the discovered structure was built to catch. 🔨 Build it yourself

    
Self-Discover is not a new architecture — it's three meta-prompts run once per task type, over a fixed seed pool of reasoning modules. Stage 1 SELECT picks the relevant modules, stage 2 ADAPT rewrites them for your task, stage 3 IMPLEMENT composes them into a JSON reasoning structure. That structure is discovered once; every actual problem instance is then solved cheaply by filling the structure in. Here's the whole pipeline.

    
    
      
  1. 
        
    1
    The seed pool of reasoning modules
    
        
    Start from the fixed menu of ~39 generic, task-agnostic reasoning modules from the paper — "break the problem into subproblems", "identify the key assumptions", "consider alternatives", "critical thinking", "step by step plan", and so on. This pool is the raw material; Self-Discover never invents new modules, it only chooses, rewords, and arranges these.
    
        
    const MODULES = [
  "How can I break this problem into smaller parts?",
  "What are the key assumptions underlying this problem?",
  "How can I simplify the problem so it is easier to solve?",
  "What are the alternative perspectives on this problem?",
  "Critical thinking: question assumptions, weigh evidence.",
  "Is this an analytical problem requiring computation?",
  "Let's make a step-by-step plan with good notation.",
  /* ...~39 generic reasoning modules total... */
];
    
      
    2. 
      
  2. 
        
    2
    Meta-prompt 1 — SELECT the relevant modules
    
        
    Show the model the task (and the whole pool) and ask it to pick the handful of modules that are useful here. Most modules are irrelevant to any given task; the point of SELECT is to throw them away so the eventual structure stays short and focused. One LLM call, returns a small list of module names.
    
        
    const selectPrompt =
 `Select the reasoning modules that are crucial to solve
  the task below. Do not elaborate — just list them.

  All modules:
  ${MODULES.join("\n")}

  Task: ${task}`;

const selected = await llm(selectPrompt);  // a short list
    
      
    3. 
      
  3. 
        
    3
    Meta-prompt 2 — ADAPT them to the task
    
        
    The generic modules are phrased for any problem. ADAPT rewrites each selected module so it speaks about this task's entities and numbers — "identify the key assumptions" becomes "a bus can't be half-hired, so round the bus count up". This is what makes the reasoning concrete instead of abstract.
    
        
    const adaptPrompt =
 `Rephrase each selected reasoning module so it is specific
  to the task. Keep the same order.

  Selected modules: ${selected}
  Task: ${task}`;

const adapted = await llm(adaptPrompt);  // task-specific wording
    
      
    4. 
      
  4. 
        
    4
    Meta-prompt 3 — IMPLEMENT the reasoning structure
    
        
    Now compose the adapted modules into an actionable, ordered plan — a small JSON key-value structure with a slot for each reasoning step. This structure is the discovered artifact: a bespoke reasoning template for this whole class of task, with empty fields waiting to be filled.
    
        
    const implementPrompt =
 `Turn the adapted modules into a step-by-step reasoning
  structure as JSON keys (leave the values empty to be
  filled while solving).

  Adapted modules: ${adapted}
  Task: ${task}`;

const structure = await llm(implementPrompt);
// e.g. { identify_quantities, state_assumption,
//        subtasks, compute_each, combine, final_answer }
    
      
    5. 
      
  5. 
        
    5
    SOLVE — fill the structure (reuse it per instance)
    
        
    Stages 1-3 run once. To solve an actual problem, hand the model the empty structure plus the specific instance and ask it to fill every field in order. Because the plan already lists the key assumption and every subtask, the model can't skip the step that plain CoT misses. The same structure is reused for every instance of the task — the discovery cost is amortised.
    
        
    function solve(instance) {
  return llm(`Follow this reasoning structure to solve the
    task. Fill in every field, then give final_answer.

    Structure: ${JSON.stringify(structure)}
    Problem: ${instance}`);
}
// discover ONCE, then:
for (const inst of manyInstances) solve(inst);
    
      
    6. 
      
  6. 
        
    6
    Know the limits — and when to reach for it
    
        
    Self-Discover shines on diverse, reasoning-heavy task suites where no single chain fits all — its edge over plain CoT grows as tasks vary. It costs three extra meta-calls up front, so it's overkill for a single one-off question (just use Plan-and-Solve). And the discovered structure is only as good as the pool and the SELECT step; a wrong plan executes cleanly to a wrong answer, so evaluate on held-out instances before trusting it.
    
        
    // Cost model:
//  SELECT + ADAPT + IMPLEMENT  → 3 calls, ONCE per task type
//  SOLVE(instance)             → 1 call per instance (reused)
//
// Use when: many instances of a varied, reasoning-heavy task.
// Skip when: a single ad-hoc question → Plan-and-Solve (Day 24).
