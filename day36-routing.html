🧭 Prompt Routing — instead of one bloated do-everything prompt that must handle billing, bugs and chit-chat all at once, first classify the incoming message, then dispatch it to the small specialized prompt (or chain) built for that case; a cheap router LLM picks the handler — and when it isn't sure, it falls back instead of guessing

Don't make one prompt handle every kind of request. Classify the input first — then watch the router pick the specialist built for it.

A single do-everything prompt that must answer billing questions, debug code, and make small talk is carrying every instruction and every tool at once — so it's bloated, expensive, and only mediocre at each job. Worse, it can't tell you when a message is ambiguous: it just picks an interpretation and confidently runs with it. Prompt Routing is a two-move pattern: a small, cheap router LLM first classifies the message into one of a fixed set of routes, then the input is dispatched to the focused specialist prompt (or chain) for that route. The router also returns a confidence — and if it's low or the top routes tie, the request goes to a fallback that asks a clarifying question or hands off to a human, rather than guessing. Below: pick or type a query, then flip between the single do-everything prompt and the router. Step through the router — classify, then dispatch — and watch it choose (including the fallback). Everything is scripted/deterministic client-side (no live model), but faithful to how a real router runs.

🔑 Routing turns one impossible prompt into a switchboard of specialists — plus an escape hatch. A cheap classifier reads the message once and decides which handler should take it; each handler is a small, sharp prompt (with only its own instructions and tools), so it answers its narrow job far better than a bloated generalist could — and often cheaper, since no single call has to carry everything. The real superpower is the confidence gate: when the router is unsure or two routes tie, it doesn't bluff — it falls back to a clarifying question or a human. The costs are real: the router adds one hop of latency, and a misroute sends the message to the wrong specialist, so keep the route set small, keep a fallback, and use a cheap model for the classify step.

Prompt routing = classify the input into one of a fixed set of routes, then dispatch it to the specialized prompt/chain for that route; gate on confidence and fall back when unsure. Better, cheaper answers per case than one do-everything prompt — at the price of a classify hop and the risk of a misroute.

🔨 Build it yourself

    
A router is just a tiny classifier plus a lookup table of specialists. No framework required. Here is the whole thing — a classify → gate on confidence → dispatch router with a fallback path, and the do-everything prompt it replaces for comparison.

    
    
      
  1. 
        
    1
    The do-everything prompt you're replacing
    
        
    First, the thing that almost works: one prompt that handles every kind of request. It's tempting because it's a single call — but it has to carry billing rules, debugging know-how, a chat persona and every tool at once, so it's bloated and pricey and mediocre at each job. And it can never say "I'm not sure which of these you meant" — it just picks one and runs.
    
        
    // ❌ One prompt for billing + code + chat + every tool. Jack of all trades.
async function doEverything(msg) {
  return llm(`You handle BILLING (refunds, charges, invoices),
             CODE bugs (errors, stack traces, APIs), AND general chat.
             You have every tool. Read the message and just do
             whatever is needed.\n\nUser: ${msg}`);
}
// bloated + pricey, only OK at each, and it can't flag "which did you want?"
    
      
    2. 
      
  2. 
        
    2
    Define the routes + their specialist prompts
    
        
    The heart of routing is a small registry: one entry per route, each a focused specialist with only its own system prompt and its own tools. The billing specialist never carries debugging instructions; the debugger never sees the refund tool. Small, sharp prompts are reliable and cheap.
    
        
    // ✅ Each route = one small, focused specialist. No cross-contamination.
const ROUTES = {
  billing: { sys: "Billing specialist. Tools: ledger.lookup, refund.issue." },
  code:    { sys: "Senior engineer. Diagnose stack traces, give a concrete fix." },
  chat:    { sys: "Friendly general assistant for greetings & how-tos." },
};
const runSpecialist = (route, msg) =>
  llm(ROUTES[route].sys + "\n\nUser: " + msg);   // one narrow, cheap call
    
      
    3. 
      
  3. 
        
    3
    The router — classify into ONE route + confidence
    
        
    The router is a tiny classifier prompt whose only job is to pick a label. Constrain it to the fixed set of routes, ask for machine-readable JSON with a confidence and a short reason, and give it an explicit "fallback" option for when nothing fits. Use a small, cheap, fast model here — this call runs on every request.
    
        
    // ✅ The router: message -> one route + confidence. Small/cheap model.
async function classify(msg) {
  const out = await llm(
    `Classify the message into ONE of: billing | code | chat.
     Reply as JSON: {"route": "...", "confidence": 0-1, "reason": "..."}.
     If none fit or you're unsure, use route "fallback".
     Message: ${msg}`);
  return JSON.parse(out);            // { route, confidence, reason }
}
    
      
    4. 
      
  4. 
        
    4
    Dispatch — gate on confidence, else fall back
    
        
    Dispatch is a lookup — but gate it first. If the router returned "fallback", or its confidence is under your threshold, do not guess a specialist: ask a clarifying question or escalate to a human. This confidence gate is the whole safety story — it's what stops a router from confidently sending an ambiguous message to the wrong expert.
    
        
    // ✅ Dispatch to the chosen specialist — but gate on confidence first.
const CONF_MIN = 0.6;
async function dispatch({ route, confidence }, msg) {
  if (route === "fallback" || confidence < CONF_MIN)
    return clarifyOrEscalate(msg);          // the safety net — never guess
  return runSpecialist(route, msg);
}
    
      
    5. 
      
  5. 
        
    5
    Put it together — classify → gate → dispatch
    
        
    The whole router is two calls: classify, then dispatch. Keep the decision around — the route, confidence and reason make routing auditable, so when an answer is off you can see whether it was a misroute (wrong route chosen) or a bad specialist answer. A route can itself be a whole chain from Day 35, so routing and chaining compose cleanly.
    
        
    // ✅ The whole router: classify -> gate -> dispatch (or fall back).
async function handle(msg) {
  const decision = await classify(msg);          // 1 cheap classify call
  console.log(decision.route, decision.confidence, decision.reason); // audit
  return dispatch(decision, msg);                // 1 focused specialist call
}
// low confidence or a tie -> clarifyOrEscalate, never a blind guess.
// a route's handler can be a full extract->transform->summarize CHAIN.
    
      
    6. 
      
  6. 
        
    6
    Know when to route — and the limits
    
        
    Route when incoming inputs fall into distinct kinds that each need different handling (different instructions, tools, or a different chain). For one uniform kind of input, a router is just an extra hop. Mind the limits: the classify call adds latency and a bit of cost (use a cheap model), a misroute propagates into a confidently wrong answer, and too many routes get hard to maintain — so keep the set small, keep a fallback, and log routes to catch drift. Routing is a core agent building-block; next you'll fan one input out to many prompts at once.
    
        
    // Decide before you build:
if (inputsSplitIntoDistinctKinds && eachNeedsDifferentHandling) {
  useRouting();        // classify -> specialist (or specialist chain)
} else if (oneUniformKindOfInput) {
  useSinglePrompt();   // a router would only add a hop
}
// always keep a fallback; a misroute = a confidently wrong answer.

The trade-offs are a classify hop of latency and the risk of a misroute — so keep the route set small, keep a fallback, and use a cheap model to classify. 👉 Next up — Parallelization: fan the same input out to several prompts at once and aggregate their results (sectioning + voting).
