🖩 Program-Aided Language Models (PAL) — write the code, let a real interpreter compute the answer

Don't make the model do the arithmetic. Make it write a program.

A language model predicts the next token, so when a problem needs exact arithmetic or bookkeeping it produces a plausible-looking number rather than a computed one — and plain Chain-of-Thought often slips a step: it treats compound interest as simple interest, rounds a conversion factor, or miscounts letters. Program-Aided Language Models (PAL) split the job in two: the model does the reasoning in code — it emits an actual program with named variables and real operations — and a deterministic interpreter does the computing. Press ▶ and watch the same problem solved both ways: CoT reasons in words and lands on the wrong number, while PAL emits a program that a real JavaScript interpreter runs live in your browser and returns the exact answer. The PAL result below is not scripted — it is whatever the code actually computes when executed.

🔑 The split is the whole trick. Chain-of-Thought asks one system — the language model — to do both the reasoning and the calculation, and calculation is exactly the part token-prediction is worst at. PAL keeps the reasoning inside the model (it decides what to compute and writes the program) but hands the calculation to an interpreter that is exact by construction. The model never states the final number; it states the method, and the runtime produces the number. That is why the PAL column above is always right and always reproducible: run it a hundred times and the interpreter returns the same value, because arithmetic in a real runtime is deterministic.

Toggle between the three problems — compound interest, unit conversion, letter counting. Each is a case where CoT tends to slip a single step: it drops the compounding, rounds a conversion factor, or miscounts a repeated letter. In every case PAL emits a short program and the browser really runs it — the answer you see in the PAL column is the interpreter's return value, computed at that moment, not a stored string.

🔨 Build it yourself

    
PAL is a three-move pipeline: prompt the model for code, run the code in a real interpreter, and return the interpreter's result as the answer. The model handles the reasoning — turning a messy word problem into named variables and operations — and the runtime handles the exact computation. Here is the whole loop, plus how to make running model-written code safe.

    
    
      
  1. 
        
    1
    Write a PAL prompt with code exemplars
    
        
    Show the model a couple of examples where a word problem is turned into a program, not a prose answer. The reasoning lives in comments and variable names; the last line produces the result. This few-shot pattern teaches the model to answer in code.
    
        
    const PAL_PROMPT = `Solve the problem by writing a program.
Put your reasoning in # comments and variable names.
Do not state the final number yourself — end by printing it.

# Q: A store had 12 apples, sold 8, then got a delivery of 20.
apples = 12
apples = apples - 8       # sold
apples = apples + 20      # delivery
print(apples)

# Q: {{QUESTION}}
`;
    
      
    2. 
      
  2. 
        
    2
    Ask for code only — reasoning goes in the program
    
        
    Send the problem and get back a program. The model still does the hard part — deciding what to compute — but expresses it as variables and operations instead of doing the mental math. Nudge it to emit a fenced code block so extraction is trivial.
    
        
    function palPrompt(question) {
  return PAL_PROMPT.replace("{{QUESTION}}", question);
}
const reply = await llm(palPrompt(
  "Invest $2,500 at 7% compounded annually for 8 years. Final balance?"
));
// reply contains a program:
//   principal = 2500
//   rate = 0.07
//   for y in range(8): principal *= (1 + rate)
//   print(round(principal, 2))
    
      
    3. 
      
  3. 
        
    3
    Extract the program from the response
    
        
    Pull the code out of the reply — usually the contents of the fenced block. Everything else the model said is just narration; the program is the artifact you execute.
    
        
    function extractCode(reply) {
  const m = reply.match(/```(?:python|js)?\n([\s\S]*?)```/);
  return m ? m[1] : reply;   // fall back to the whole reply
}
const program = extractCode(reply);
    
      
    4. 
      
  4. 
        
    4
    Run it in a real interpreter
    
        
    Hand the program to an actual runtime — a Python subprocess, a sandboxed JS VM, a code-interpreter tool — and capture what it prints. This is the step that makes the arithmetic exact: the runtime, not the model, produces the number. (The LOOK tab does exactly this with the browser's own JS engine.)
    
        
    // Node example: run the model's Python in a subprocess
import { execFileSync } from "node:child_process";
function runPython(src) {
  return execFileSync("python3", ["-c", src], {
    timeout: 2000, encoding: "utf8"
  }).trim();
}
const result = runPython(program);   // e.g. "4295.47"
    
      
    5. 
      
  5. 
        
    5
    Return the interpreter's result as the answer
    
        
    The value the program printed is the answer — you skip having the model read the number back and re-transcribe it (another place it can slip). Optionally feed the result to a final call that phrases it in a sentence for the user.
    
        
    const answer = result;                 // exact, computed value
// optional: wrap it in prose for the user
const reply = await llm(
  `The computed answer is ${answer}. State it in one sentence.`);
    
      
    6. 
      
  6. 
        
    6
    Sandbox the code — and fall back when it fails
    
        
    You are running text a model wrote, so never eval it in your own process with full privileges. Execute it in an isolated sandbox with no network or filesystem access, a memory cap and a hard timeout. Validate the output shape, and if the program errors or hangs, retry once or fall back to plain CoT.
    
        
    async function pal(question) {
  const program = extractCode(await llm(palPrompt(question)));
  try {
    const out = await sandbox.run(program, {   // isolated VM
      timeoutMs: 2000, memoryMb: 128,
      network: false, filesystem: false
    });
    return validate(out);            // check it's a number/JSON
  } catch (e) {
    return await llm(cotPrompt(question));   // fall back to CoT
  }
}

You learned Program-Aided Language Models: instead of asking the model to compute the answer, you ask it to write a program and let a deterministic interpreter run it. The reasoning stays with the model; the arithmetic and bookkeeping move to a runtime that is exact and reproducible — which is exactly where token-prediction is weakest. 👉 Tomorrow — Day 28: Rephrase-and-Respond (RaR) — have the model restate your question in its own words before answering, so it locks onto what you actually meant.
