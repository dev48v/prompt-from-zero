PromptFromZero · Day 42

🌲 Maieutic Prompting — instead of trusting one reasoning chain, make the model explain a claim, then recursively justify every premise into a tree of explanations. Probe each node for logical integrity, detect the contradictions between them, and prune the inconsistent subtree — the consistent survivors give an answer that holds even when the model is wrong at the leaves.

Grow a maieutic tree for a tricky question — then watch logic, not a vote, prune away the wrong answer.

Ask a model a deceptively simple question — "Is a whale a fish?" — and a single reasoning chain can wander into the intuitive-but-wrong answer ("it lives in water and swims, so it's a fish"). Maieutic prompting doesn't ask for one answer; it asks the model to explain the claim both ways and then recursively justify every premise, growing a tree of explanations. Some premises are logically integral — the model defends them consistently no matter how you ask ("a whale is a mammal", "mammals are not fish"). Others are shaky — asserted here but easily flipped ("anything that lives in water is a fish"). Below: press ▶ Grow the tree to build it, then 🔍 Detect contradictions & prune. The tree is a real data structure in JavaScript and the contradiction check is a real graph traversal — it finds where a shaky premise collides with the integral facts, marks that subtree red, prunes it, and reads the answer off the consistent survivors. 🔑 Maieutic prompting = reason as a tree you can audit, not a chain you must trust. Ask the model to explain a proposition both ways, recurse into every premise to grow a tree of explanations, keep the logically integral nodes (defended consistently no matter how you ask) as near-hard constraints, and resolve the whole thing as a weighted-satisfiability problem: wherever a shaky premise contradicts the integral facts, prune that subtree. The surviving consistent set is the answer — and it comes with its own rationale. Unlike chain-of-thought (one unchecked line) or self-consistency (a vote that can be confidently wrong), maieutic decides by logical consistency, so it stays right even when the model is unreliable at the leaves. It costs many more calls, so spend it on hard true/false, commonsense, and logical questions where correctness and an auditable reason matter.

🔨 Build it yourself

    
A maieutic pipeline is a loop of ordinary prompt calls wrapped around a tiny bit of graph logic: explain both ways → recurse into premises → test each node's integrity → build a consistency graph → solve MAX-SAT → prune → read the answer. Here it is end to end — the prompts, the tree, the both-directions integrity probe, the relation graph, the weighted solve that prunes contradictions, and when it's worth the extra calls.

    
    
      
  1. 
        
    1
    Ask for an explanation — both ways
    
        
    Don't ask "is Q true?". Ask the model to explain Q as true, and separately to explain Q as false. An explanation exposes the premises the answer rests on — and getting both sides is what later lets you test whether a claim survives being argued against.
    
        
    // STEP 1 — explain the proposition BOTH ways (abductive prompting).
const Q = "A whale is a fish.";

const proTrue  = await llm(`${Q}\nThis is TRUE because`);   // argue for
const proFalse = await llm(`${Q}\nThis is FALSE because`);  // argue against

// each explanation names the premises its conclusion depends on:
//   proTrue  -> "it lives in water and swims, and water animals are fish"
//   proFalse -> "it is a mammal, and mammals are not fish"
    
      
    2. 
      
  2. 
        
    2
    Recurse into the premises — grow the tree
    
        
    Treat each premise as a new proposition and explain it both ways too. Repeat to a fixed depth (or until a premise is self-evident). The result is a tree: the root is the question, and every node is a sub-explanation that bottoms out in simple, checkable claims.
    
        
    // STEP 2 — recurse: every premise becomes a child proposition.
async function grow(prop, depth) {
  const node = { prop, children: [] };
  if (depth === 0) return node;                 // stop at max depth
  for (const side of ["TRUE", "FALSE"]) {
    const expl = await llm(`${prop}\nThis is ${side} because`);
    for (const premise of extractPremises(expl)) {
      node.children.push(await grow(premise, depth - 1)); // <-- recurse
    }
  }
  return node;                                   // a real explanation TREE
}
const tree = await grow(Q, 2);
    
      
    3. 
      
  3. 
        
    3
    Test each node's logical integrity
    
        
    This is the key filter. For each proposition, ask the model to defend it as true AND to defend it as false. If it firmly holds one side and contradicts itself trying to argue the other, the node is logically integral — a trustworthy anchor. If it will happily argue both, it's shaky and gets a low weight.
    
        
    // STEP 3 — a node is "integral" if the model can only defend ONE side.
async function isIntegral(prop) {
  const affirm = await llm(`${prop}\nWhy is this TRUE?`);
  const deny   = await llm(`${prop}\nWhy is this FALSE?`);
  // integral = the model firmly commits one way and cannot honestly
  // argue the other (it contradicts itself or refuses).
  return firmlyHolds(affirm) !== firmlyHolds(deny); // true = trustworthy
}
// "a whale is a mammal"          -> integral (only defends TRUE)
// "all water-dwellers are fish"  -> NOT integral (flips under pressure)
    
      
    4. 
      
  4. 
        
    4
    Build the consistency graph (relations)
    
        
    Ask the model (or a natural-language-inference check) whether pairs of propositions are consistent or contradictory. Those become edges. Now you have a graph: nodes weighted by integrity, edges marking which claims can and cannot both be true.
    
        
    // STEP 4 — relations between propositions become graph edges.
function relation(a, b) {
  // "consistent" (can both hold) or "contradict" (cannot both be true)
  return llmNLI(a, b); // e.g. entailment / contradiction / neutral
}
// edge found:
//   "all water-dwellers are fish"  ⨯  "mammals are not fish"  -> CONTRADICT
const edges = [
  { a: "all water-dwellers are fish", b: "mammals are not fish", type: "contradict" }
];
    
      
    5. 
      
  5. 
        
    5
    Solve MAX-SAT and prune the inconsistent subtree
    
        
    Turn it into weighted satisfiability: each node is a variable, integral nodes are near-hard constraints (high weight), shaky ones are soft. Find the truth assignment that satisfies the most weighted constraints. The false premise loses — it's out-weighed by the integral chain it contradicts — so you prune it and the explanation built on top of it.
    
        
    // STEP 5 — weighted-SAT: keep the max-consistent set, prune the rest.
function resolve(nodes, edges) {
  const weight = n => n.integral ? 100 : 5;      // integrity = constraint weight
  // for each contradiction edge, the side backed by MORE integral
  // weight wins; the loser (and its dependent subtree) is pruned.
  for (const e of edges.filter(e => e.type === "contradict")) {
    const loser = weight(e.a) < weight(e.b) ? e.a : e.b;
    prune(loser);        // mark loser + descendants inconsistent (red)
    pruneAncestorsThatDependedOn(loser);         // the branch collapses
  }
  return nodes.filter(n => n.state === "active"); // consistent survivors
}
    
      
    6. 
      
  6. 
        
    6
    Read the answer — and know when to spend the calls
    
        
    The surviving consistent set all points one way — that's your answer, and the survivors are its rationale (an auditable trail, not a black box). Maieutic shines on hard true/false, commonsense, and logical questions where a single chain is brittle. It costs many model calls (explain both ways × recurse × integrity probes), so save it for when correctness and an explanation matter. Next up you'll switch topics to system prompts: the standing persona-and-rules turn that shapes everything the model says.
    
        
    // STEP 6 — the survivors give the answer AND the reason.
const survivors = resolve(nodes, edges);
const answer = majoritySide(survivors);   // all consistent -> "NOT a fish"
console.log(answer, survivors.map(n => n.prop)); // built-in audit trail

// WHEN TO USE MAIEUTIC:
//  - hard true/false, commonsense, logical questions (one chain is brittle)
//  - when you need an AUDITABLE reason, not just an answer
//  - when the model is unreliable at the leaves but its logic is checkable
// WHEN NOT TO:  simple lookups, open-ended generation, or latency/cost-
//               sensitive paths — it multiplies calls (explain × recurse × probe)
