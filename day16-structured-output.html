🧱 Structured Output — Make the LLM Return Real JSON

Your code needs typed JSON — not a paragraph that happens to contain some.

A model talks to code, not just people, and code needs one exact shape. Just asking for JSON in the prose works until the day it wraps the object in chatter, a ```json fence, a trailing comment, or the wrong type — and JSON.parse throws. Binding the model to a schema (function calling / structured outputs / JSON mode) forces it to emit exactly your fields, with the right types, every time. 🔨 Bind a model to a schema and get typed JSON back

    
Define the shape you want as a schema, hand it to the model as a tool or a response_format, call the model, then parse and validate the result. If validation fails, retry with the error fed back; once it passes, you hold a typed object your code can trust.

    
    
      
  1. 1
    Define the schema you actually want
    import { z } from "zod";

// Describe the exact shape. Types, enums and required fields
// are the contract the model must satisfy — not a suggestion.
const FlightBooking = z.object({
  name:       z.string(),
  from:       z.string(),               // origin city
  to:         z.string(),               // destination city
  date:       z.string().describe("ISO date, e.g. 2026-03-03"),
  passengers: z.number().int().min(1),
  cabin:      z.enum(["economy", "business", "first"]),
});
// One source of truth: validation AND the model both read this.
    2. 

      
  2. 2
    Turn the schema into JSON Schema
    import { zodToJsonSchema } from "zod-to-json-schema";

// Providers speak JSON Schema. Generate it from the Zod object
// so the model and your validator never drift apart.
const jsonSchema = zodToJsonSchema(FlightBooking, "FlightBooking");
// {
//   type: "object",
//   properties: { name:{type:"string"}, passengers:{type:"integer"}, ... },
//   required: ["name","from","to","date","passengers","cabin"]
// }
    3. 

      
  3. 3
    Pass it as a tool (function calling)
    // Expose a named function whose arguments ARE your schema.
// The model "calls" it by filling typed arguments — it can't
// reply with prose when a tool call is forced.
const tools = [{
  type: "function",
  function: {
    name: "book_flight",
    description: "Extract a flight booking request.",
    parameters: jsonSchema,
  },
}];

const res = await client.chat.completions.create({
  model, tools,
  tool_choice: { type: "function", function: { name: "book_flight" } },
  messages: [{ role: "user", content: sentence }],
});
    4. 

      
  4. 4
    Or ask for JSON directly (response_format)
    // Structured Outputs: hand the schema as the response format and
// the decoder is constrained to emit ONLY conforming JSON.
const res = await client.chat.completions.create({
  model,
  messages: [{ role: "user", content: sentence }],
  response_format: {
    type: "json_schema",
    json_schema: { name: "FlightBooking", schema: jsonSchema, strict: true },
  },
});
const raw = res.choices[0].message.content;   // guaranteed JSON text
    5. 

      
  5. 5
    Parse, then validate against the schema
    // Never trust raw text. Parse, then validate so a wrong type or
// a missing field is caught here, not three layers downstream.
function parseBooking(raw) {
  const obj = JSON.parse(raw);          // syntax check
  return FlightBooking.parse(obj);      // shape + type check (throws on bad)
}
const booking = parseBooking(raw);
// booking.passengers is a real number, booking.cabin is a valid enum.
    6. 

      
  6. 6
    Retry on invalid — feed the error back
    // Even constrained models occasionally trip. Loop a couple of
// times, handing back the exact validation error each time.
async function extract(sentence, tries = 3) {
  let err = "";
  for (let i = 0; i < tries; i++) {
    const raw = await callModelWithSchema(sentence, err);
    try { return FlightBooking.parse(JSON.parse(raw)); }
    catch (e) { err = `Previous output was invalid: ${e.message}. Fix it.`; }
  }
  throw new Error("Could not get valid structured output.");
}
    7. 

      
  7. 7
    Constrain choices with enums & required
    // The tighter the schema, the less the model can improvise.
const Ticket = z.object({
  priority: z.enum(["low", "medium", "high", "urgent"]),  // closed set
  category: z.enum(["billing", "bug", "feature", "other"]),
  customer: z.string().email().optional(),                // optional field
  summary:  z.string().max(120),                          // bounded length
});
// Enums kill free-text drift; required vs optional makes intent explicit.
    8. 

      
  8. 8
    Use the typed object — and still trust the values
    const b = await extract(sentence);
await flightsApi.reserve({               // hand it straight to your code
  passenger: b.name, origin: b.from, destination: b.to,
  departureDate: b.date, seats: b.passengers, cabin: b.cabin,
});
// ⚠️ The SHAPE is guaranteed; the VALUES are not. Schemas stop
// "passengers: 'two'" — they don't stop a wrong city. Sanity-check
// business rules (valid airport, date in the future) yourself.
