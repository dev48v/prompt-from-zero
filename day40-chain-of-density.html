🧬 Chain-of-Density — start with a sparse summary, then rewrite it again and again at the same fixed length, packing 1–3 more salient entities into each pass. Density climbs while length stays flat: the model fuses new facts and compresses filler to make room. Denser reads richer — up to a point, after which readability suffers — so the best summary sits at a sweet spot in the middle of the chain, not at the end.

A first summary is usually too sparse. Don't make it longer — make it denser: same length, more facts, one pass at a time.

A one-shot summary tends to be entity-poor — it names a couple of things and pads the rest with filler ("remarkable detail", "an important moment"). You could just ask for "more detail", but then it gets longer and you lose the whole point of a summary. Chain-of-Density (CoD) fixes the length and iterates on the content: begin with a deliberately sparse summary, then each pass identifies 1–3 salient entities missing from it and rewrites the summary to include them without growing — by fusing new facts into existing sentences and compressing the filler that was taking up space. Entity density climbs steeply while length stays flat. Denser reads richer — but only up to a point: cram too hard and the prose turns into a comma-spliced list that's hard to read, so the best version is a sweet spot in the middle, not the maximally-dense final one. Below: pick a source article, step the chain, and watch each pass's summary, the entities it packed in, a rising density meter at flat length, and a quality curve that peaks then dips — then flip to ⚡ one-shot to see the sparse summary you'd have shipped. Everything is scripted/deterministic client-side (no live model), but faithful to how a real CoD run behaves.

🔑 Chain-of-Density = fix the length, then iteratively raise the information density. Start sparse; each pass identifies 1–3 salient entities missing from the current summary and rewrites at the same length, fusing the new facts in and compressing filler to make room. The signature is density up, length flat. But informativeness and readability pull against each other — the maximally-dense last pass is often an unreadable list — so pick the sweet spot in the middle, the richest summary a human still reads comfortably. Practically it's cheap: one prompt can emit the whole chain at once (the model returns every step in JSON), and you choose the pass you want.

🔨 Build it yourself

    
Chain-of-Density is one move layered onto plain summarization: sparse draft → [find missing entities → rewrite at SAME length]* → pick the sweet spot. The new idea is the fixed-length rewrite — you improve by raising density, never by adding words. Here is the whole thing: the sparse start, the length lock, the missing-entity step, the fuse-and-compress rewrite, the loop that tracks density, and the assembled function (including the single-prompt shortcut).

    
    
      
  1. 
        
    1
    The sparse start you're upgrading
    
        
    Begin on purpose with a deliberately sparse summary — the kind a one-shot prompt gives you. It hits the target length but is mostly filler, naming only a couple of entities. That's the floor; everything below raises the density from here without touching the length.
    
        
    // Step 0 — a sparse, filler-heavy first summary (the floor).
let summary = await llm(`
  Summarize the ARTICLE in about ${TARGET} words.
  This first version may be non-specific and use fillers.
  ARTICLE: ${article}
`);
// e.g. "A powerful new telescope released its first pictures..." (2 entities)
    
      
    2. 
      
  2. 
        
    2
    Lock the length — the whole constraint
    
        
    The one rule that makes CoD work: the length never changes. Set a target word count once and hold it every pass. This is what forces the model to earn room for new facts by cutting filler, instead of just appending — the reason density rises rather than the summary ballooning.
    
        
    // The invariant: TARGET is fixed for the entire chain.
const TARGET     = 50;   // words — held constant every rewrite
const STEPS      = 5;    // how many densification passes
const PER_STEP   = "1-3"; // new entities to add each pass
// Rule enforced in every prompt: "Keep the length the same."
    
      
    3. 
      
  3. 
        
    3
    Find the missing salient entities
    
        
    Each pass starts by asking: which important entities are in the article but not yet in the summary? Entities are the concrete nouns — people, places, dates, numbers, named things. Pick the 1–3 most salient that are missing. This targeting is what makes the added content matter instead of being random detail.
    
        
    // Identify what's salient in the source but absent from the summary.
const missing = await llm(`
  List the 1-3 MOST salient entities in the ARTICLE that are
  MISSING from the SUMMARY. Entities = people, places, dates,
  numbers, named things. Return a JSON array of short strings.
  ARTICLE: ${article}
  SUMMARY: ${summary}
`);  // e.g. ["James Webb Space Telescope", "NASA", "July 12, 2022"]
    
      
    4. 
      
  4. 
        
    4
    Rewrite = fuse + compress, same length
    
        
    The heart of the pass: rewrite the summary so it includes the new entities but stays the same length. The model does this two ways — fusion (merge a new fact into an existing clause) and compression (drop filler words, uninformative phrases, and connectives). No new sentences; you make space, you don't add space.
    
        
    // Rewrite at the SAME length — fuse the new entities, compress filler.
summary = await llm(`
  Rewrite the SUMMARY so it now covers these NEW entities,
  keeping EXACTLY the same length (~${TARGET} words).
  Make room by FUSING facts into existing sentences and
  COMPRESSING filler ("remarkable detail", "an important moment").
  Do NOT drop any entity already present.
  NEW: ${missing}
  SUMMARY: ${summary}
`);
    
      
    5. 
      
  5. 
        
    5
    Loop — keep every step, track density
    
        
    Repeat find→rewrite for a fixed number of steps, and — crucially — keep every intermediate summary. Unlike a pass/fail loop, CoD doesn't converge on one answer; it produces a ladder from sparse to dense. Track density (entities ÷ words) at each rung so you can choose later. Length stays flat the whole way; only density moves.
    
        
    // Build the full ladder — sparse to dense — and remember each rung.
const chain = [{ summary, entities: entitiesIn(summary) }];
for (let s = 1; s <= STEPS; s++) {
  const missing = await findMissing(article, summary);
  summary       = await densify(summary, missing, TARGET); // same length
  chain.push({
    summary,
    entities: chain[s-1].entities.concat(missing),
    density: (chain[s-1].entities.length + missing.length) / words(summary)
  });
}
// density climbs each step; words(summary) barely changes.
    
      
    6. 
      
  6. 
        
    6
    Pick the sweet spot — and the 1-prompt shortcut
    
        
    The last rung is the densest, but usually not the best — it's a comma-spliced list. Informativeness rises while readability falls, so choose the sweet spot where they balance (in the CoD paper, humans preferred the middle passes). And you don't need N round-trips: one prompt can emit the entire chain as JSON, and you pick a rung. Next you'll switch gears to delivery: streaming tokens as they're generated.
    
        
    // Whole chain in ONE call, then choose the readable-but-rich rung.
const chain = JSON.parse(await llm(`
  Produce ${STEPS} increasingly dense summaries of the ARTICLE.
  Each is ~${TARGET} words (SAME length). Each adds ${PER_STEP} missing
  salient entities vs the previous, fusing + compressing to fit.
  Return JSON: [{ step, summary, addedEntities }].
  ARTICLE: ${article}
`));
const pick = chain[Math.round(chain.length * 0.6)]; // middle-ish sweet spot
return pick.summary;   // richest summary a human still reads easily
