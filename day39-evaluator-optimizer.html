📈 Evaluator–Optimizer — one model (the optimizer) drafts, then a separate evaluator scores that draft against an explicit rubric and hands back concrete feedback; the optimizer revises, and the loop repeats until the output clears the bar or hits a max-round cap. Quality is earned through iteration — not hoped for in a single shot, and not the same model vaguely critiquing itself (Day 30).

Don't ship the first draft and hope. Let a separate evaluator grade it against a rubric, hand back feedback, and loop the optimizer until the output earns a pass.

A single-shot prompt gives you one draft and no idea whether it's good — you find out in production. Evaluator–Optimizer closes the loop with two distinct roles: the optimizer generates a draft, then a separate evaluator scores it against an explicit rubric (weighted criteria, a numeric score) and returns concrete, actionable feedback. The optimizer folds that feedback into a revision, the evaluator grades again, and the cycle repeats — until the score clears a pass threshold, or the loop hits a max-round cap and returns the best draft so far. This is not Day-30 self-refine: there the same model informally critiques its own work; here the evaluator is a separate role with named criteria and a numeric bar, so "good enough" is measured, not guessed, and the score is comparable round over round. Below: pick a task, run the loop, and watch each round's draft, per-criterion scores, and feedback — the score climbing toward the bar — then flip to ⚡ single-shot to see what you'd have shipped without the loop. Everything is scripted/deterministic client-side (no live model), but faithful to how a real evaluator–optimizer loop runs.

🔨 Build it yourself

    
Evaluator–Optimizer is one move added to a plain generator: optimize → [evaluate → revise]* until pass|cap. The new idea over Day 37/38 is a second role — an evaluator that scores against a rubric and returns feedback — plus a loop with clear stopping criteria. Here is the whole thing: the single-shot you're upgrading, the evaluator, the rubric, the revise step, the loop, and the assembled function.

    
    
      
  1. 
        
    1
    The single-shot you're upgrading
    
        
    First, what almost works: generate one draft and ship it. There's no quality gate — the output might be excellent or garbage, and you have no way to tell before it's in front of a user. Everything below exists to replace that blind hope with a measured loop.
    
        
    // ❌ Single-shot: draft once and ship. No quality gate.
const draft = await llm(`Write a tagline for: ${brief}`);
return draft;   // good? bad? no idea — you find out in production.
    
      
    2. 
      
  2. 
        
    2
    The evaluator — a SEPARATE role that scores vs a rubric
    
        
    The core new idea. A distinct evaluator call reads the draft and grades it against explicit criteria, returning a structured verdict: a numeric score, a pass flag, and concrete feedback. This is what separates it from Day-30 self-refine — a separate role with named criteria, not the same model informally second-guessing itself.
    
        
    // ✅ A distinct evaluator: scores the draft against explicit criteria.
const verdict = await llm(`
  You are a strict evaluator. Score the DRAFT against this rubric
  (0-100 total) and give concrete, actionable feedback.
  Return JSON: { "score": n, "pass": bool, "feedback": "..." }
  RUBRIC: ${JSON.stringify(rubric)}
  DRAFT:  ${draft}
`);
const { score, pass, feedback } = JSON.parse(verdict);
// separate role + explicit rubric — not the same model vibing (Day 30).
    
      
    3. 
      
  3. 
        
    3
    The rubric & the bar — make "good" measurable
    
        
    The rubric is the spec. List the weighted criteria that define quality for this task, set the threshold the score must clear, and set a max-round cap so a stubborn task can't loop forever. Explicit criteria make the score comparable round over round — you can watch it climb.
    
        
    // ✅ Make "good" explicit — the rubric is the contract.
const rubric = [
  { name: "Clarity",          max: 20 },
  { name: "Concrete benefit", max: 20 },
  { name: "Emotional hook",   max: 20 },
  { name: "Punchy (<=8 words)", max: 20 },
  { name: "Distinctive",      max: 20 },
];
const PASS = 88;       // the bar the score must clear
const MAX_ROUNDS = 5;  // and the budget that stops runaway loops
    
      
    4. 
      
  4. 
        
    4
    The revise step — condition the next draft on feedback
    
        
    The optimizer's revision is conditioned on the evaluator's feedback — that hand-off is the engine of the whole pattern. Give it the current draft and the specific feedback, and ask it to improve while keeping what already works. Without feedback threading through, you're just re-rolling the dice.
    
        
    // ✅ The optimizer revises USING the evaluator's feedback.
async function revise(draft, feedback) {
  return await llm(`
    Improve the DRAFT using the FEEDBACK. Keep what works.
    FEEDBACK: ${feedback}
    DRAFT:    ${draft}
  `);
}
// feedback -> revision is the engine; without it, you're re-rolling.
    
      
    5. 
      
  5. 
        
    5
    The loop — evaluate, revise, track the best, stop
    
        
    Now wire it into a loop with two stopping criteria: return the moment a draft passes, and never exceed MAX_ROUNDS. Track the best draft seen, so that if the budget runs out you return the best-so-far rather than a failure. This is the difference between a controlled loop and an infinite one.
    
        
    // ✅ Loop until it passes OR the budget runs out — keep the best.
let draft = await optimize(brief), best = null;
for (let round = 1; round <= MAX_ROUNDS; round++) {
  const { score, pass, feedback } = await evaluate(draft, rubric);
  if (!best || score > best.score) best = { draft, score };
  if (pass) return draft;                 // cleared the bar — done
  draft = await revise(draft, feedback);  // else fold feedback in, retry
}
return best.draft;   // budget spent — return best-so-far, not a crash
    
      
    6. 
      
  6. 
        
    6
    Put it together — and know the limits
    
        
    The whole pattern: optimize → [evaluate → revise]* until pass|cap. Reach for it when you can articulate what "good" means and quality beats latency — refining copy, hardening code, tuning a query. Mind the limits: it costs ~2 calls per round; a weak rubric means it optimizes the wrong thing; scores often plateau, so cap the rounds and watch for diminishing returns; and the evaluator can be wrong — a bad grader is worse than none. Next you'll let the model run its own loop entirely.
    
        
    // ✅ Evaluator–Optimizer = optimize -> [evaluate -> revise]* until pass|cap.
async function evalOptimize(brief) {
  let draft = await optimize(brief);
  for (let r = 1; r <= MAX_ROUNDS; r++) {
    const v = await evaluate(draft, rubric);    // 1 evaluator call
    if (v.pass) return draft;                    // stop: cleared the bar
    draft = await revise(draft, v.feedback);     // 1 optimizer call
  }
  return draft;   // cost ~ 2 calls/round; the rubric is everything.
}
