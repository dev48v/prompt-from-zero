PromptFromZero · Day 30

♻️ Self-Refine — the model drafts an answer, critiques its own draft, then rewrites using that critique, looping until the gains flatline — one model as generator, critic, and reviser, with no external tools

Great writing is rewriting — and one model can do both halves. Draft, then turn the same model into its own harshest editor, and loop.

Ask a model for something once and you get exactly one pass: a first draft — fluent, plausible, and almost always improvable. A function that forgets the empty-input case; a paragraph padded with “really great”; an email with no deadline. The model often knows better — it can spot those flaws in a second if asked — but a single forward pass never stops to check. Self-Refine (Madaan et al., 2023) bakes the rewrite instinct into the prompt: the same model produces a draft, then switches hats and CRITIQUES its own output — listing concrete, specific flaws — then switches again and REWRITES the draft to fix exactly those flaws. Critique the new version, refine again, and repeat until the self-critique stops finding anything worth changing and the quality gain flattens. No tests, no compiler, no human, no second model — one model playing generator, critic, and reviser in turn. Pick a task and step through the loop below: watch each critique name real problems, watch the next version address exactly those, and watch the quality meter climb then plateau. Everything is scripted and deterministic — no live model — so the trajectory is identical every run.

🔑 The critique is the engine — and the same model supplies it. Nothing external judged these drafts: no unit test, no editor, no second model. The model wrote the draft, then — pointed at its own finished output with the single narrow job “find what's wrong here” — surfaced concrete flaws it hadn't avoided the first time, and the refine step put that knowledge to work. Recognizing a weakness is an easier task than avoiding it under next-token pressure, which is exactly why a second look helps even though the weights never changed. The loop stops itself when the self-critique runs dry and each new round barely moves the needle — the plateau you see on the right.

Draft → self-critique → refine → self-critique → refine … until the critique finds nothing actionable and the marginal quality gain drops below threshold. Most of the lift lands in the first one or two refinements; returns diminish fast — so a small round cap captures nearly all the benefit.

🔨 Build it yourself

    
Self-Refine is three prompts and a loop. Generate a draft, critique it (specific, itemized, actionable — this is the load-bearing step), then refine it against that critique. Wrap those in a loop with a stopping criterion, ground the critic in a rubric so it doesn't rubber-stamp, and know when to hand the feedback job to an external oracle instead. Here is each piece.

    
    
      
  1. 
        
    1
    Prompt #1 — Generate the draft
    
        
    The first pass is just the task, done once. Expect first-draft quality: fluent and plausible, but carrying the small errors, omissions, and clumsy phrasings any first draft has. This is the raw material the loop will improve — don't over-engineer it.
    
        
    const GENERATE = `${task}`;      // just do the task, one pass

let draft = await llm(GENERATE);
// draft = first-draft quality: fluent, plausible, improvable.
// the model often KNOWS better -- it just never stopped to check.
    
      
    2. 
      
  2. 
        
    2
    Prompt #2 — Critique it (specific, actionable, itemized)
    
        
    This is the prompt that carries the whole technique. Point the model at its own finished output and give it the single narrow job of finding flaws. Demand specific, actionable, quoted problems — “make it better” yields nothing; “the loop divides by length with no empty-array guard” drives a real fix. Give it an explicit escape hatch so a genuinely-good draft can end the loop.
    
        
    const FEEDBACK = (draft) => `Here is an attempt at the task:
"""${draft}"""

List the SPECIFIC, ACTIONABLE problems with it as bullets —
correctness, missing cases, clarity, structure, tone.
Be concrete and quote the text you mean.
If it is already excellent, reply exactly: NO_ISSUES.`;

const critique = await llm(FEEDBACK(draft));
// vague critique -> vague rewrite. Insist on specifics.
    
      
    3. 
      
  3. 
        
    3
    Prompt #3 — Refine using the critique
    
        
    Hand the model both the draft and its own critique, and ask for a rewrite that addresses every point. The critique is what makes this more than a re-roll: the new version is conditioned on a concrete list of what was wrong, so it fixes those things instead of regenerating from scratch and reintroducing them.
    
        
    const REFINE = (draft, critique) => `Task attempt:
"""${draft}"""

Critique of that attempt:
"""${critique}"""

Rewrite the attempt so it fixes EVERY point in the critique.
Output only the improved version.`;

draft = await llm(REFINE(draft, critique));  // v2 addresses each flaw
    
      
    4. 
      
  4. 
        
    4
    Wire the loop — with a stopping criterion
    
        
    Alternate critique and refine until you should stop. Three stops, usually combined: the critique itself says it's done (NO_ISSUES), the marginal quality gain drops below a threshold (you're polishing noise), or you hit a hard round cap. Most of the lift is in the first one or two rounds, so a small cap is cheap insurance against an endless — and expensive — loop.
    
        
    async function selfRefine(task, { maxRounds = 3, minGain = 0.05 } = {}) {
  let draft = await llm(task);
  let prev  = score(draft);
  for (let r = 0; r < maxRounds; r++) {
    const critique = await llm(FEEDBACK(draft));
    if (critique.trim() === "NO_ISSUES") break;        // self-judged done
    const next = await llm(REFINE(draft, critique));
    const q = score(next);
    draft = next;
    if (q - prev < minGain) break;                     // marginal gain -> stop
    prev = q;
  }
  return draft;   // ~2K+1 model calls for K rounds
}
    
      
    5. 
      
  5. 
        
    5
    Ground the critic in a rubric so it can't rubber-stamp
    
        
    An open-ended “what's wrong?” invites the model to shrug and say “looks good” — an empty critique that wastes a whole round. Give the critic a rubric or checklist to grade against, and force it to justify any criterion it scores below full marks. Structured critique is more consistent, harder to hand-wave, and turns vague dissatisfaction into concrete, refinable fixes.
    
        
    const FEEDBACK = (draft) => `Grade this output against the rubric,
then list concrete fixes for EVERY criterion scored below 5/5:
- Correctness   (handles edge cases? logically sound?)
- Completeness  (anything missing or unstated?)
- Clarity       (unambiguous, concise, no padding?)
- Structure     (format / ordering fit the task?)

Output: a score per criterion, then the itemized fixes.

Attempt:
"""${draft}"""`;
// a rubric stops "looks good" rubber-stamping and empty critiques
    
      
    6. 
      
  6. 
        
    6
    Prefer an external oracle when one exists (the Reflexion move)
    
        
    Self-critique is only as sharp as the model's own eyes — it can be blind to its own errors, or even “refine” a correct answer into a wrong one. Wherever a cheap, trustworthy oracle exists — unit tests, a compiler, a type-checker, a calculator — feed that result back instead: it can't be fooled by the model's blind spots. Use self-critique for the open-ended tasks (most writing and reasoning) where no such oracle is available; the two compose.
    
        
    // when a cheap, trustworthy oracle exists, feed IT back (Reflexion-style)
let code = await llm(task);
for (let r = 0; r < maxRounds; r++) {
  const result = runTests(code);        // compiler / unit tests / linter
  if (result.pass) break;               // oracle says done -- can't be fooled
  code = await llm(REFINE(code, result.errors));
}
// self-critique only where NO oracle exists (open-ended writing/reasoning)

🎉 Day 30 of PromptFromZero done.
You learned Self-Refine: have one model draft, then critique its own draft, then rewrite using that critique, looping until the self-critique runs dry and the quality gain flattens. It works because recognizing a flaw is easier than avoiding it under next-token pressure — the critique surfaces knowledge the draft didn't use. Three prompts (generate / feedback / refine), a stopping criterion (self-judged done · marginal gain · max rounds), a rubric to stop rubber-stamping, and an external oracle wherever one exists. Cousin to Reflexion (external feedback) and Chain-of-Verification (fact-checking); costs ~2K+1 calls for K rounds. It shines on code, writing, and multi-constraint reasoning — anywhere a first draft is decent but a second look reliably makes it better. 👉 Tomorrow — Day 31: Emotion-Prompting — add emotional stakes to the prompt (“this is important to my career”) and watch the model try harder.
