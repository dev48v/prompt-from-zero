🪆 Analogical Prompting — make the model generate its own relevant worked examples before it solves, so it reasons by analogy instead of from a cold start

A student who works two similar problems first rarely blows the third. Give the model that warm-up — and let it write the examples itself.

Answering a tricky problem cold, a language model does what it does with everything: it grabs the nearest surface pattern and runs with it — and on problems with a well-known trap (a leading-digit rule, a compounding discount, a rate you have to divide out first) the nearest pattern is usually the wrong one. Few-shot prompting fixes this by pasting in solved examples — but you have to have good, relevant, correct examples on hand for every new problem. Analogical Prompting (Yasunaga et al., 2023) removes that burden: before solving, you tell the model to recall and write out a few relevant solved problems itself — “here are similar problems and how I'd solve them” — and only then tackle the target by analogy. The self-generated exemplars warm up the right reasoning pattern from the model's own pretraining. Pick a target below and press ▶: the left column answers cold and falls into the trap; the right column self-generates examples first, then solves correctly. Everything is scripted and deterministic — no live model — so the contrast is identical every run.

🔑 The examples do the work — and the model can write them itself. Nothing about the model's knowledge changed between the two columns: it always knew the leading-digit rule, that discounts compound, that rates divide out. What changed on the right is that it recalled a couple of solved problems of the same shape before touching the target, and that warm-up activated the correct method. Few-shot gives the model the same head start — but only if you already have the right examples written down. Analogical prompting is few-shot where the model curates its own exemplars, so it works even on a brand-new problem you have no examples for.

Toggle the source of the examples: Analogical = the model generates the exemplars on the spot from pretraining; Few-shot = the same examples, but you had to find and write them yourself for this exact problem. Same reasoning payoff — the difference is who curates the examples, and whether you needed a good one on hand.

🔨 Build it yourself

    
Analogical prompting is pure prompt engineering — you add one instruction telling the model to recall a few relevant solved problems before answering. Here is the core template, how to give each self-generated exemplar its own reasoning (analogical + CoT), how to control how many you get, the generated-knowledge cousin, how to guard against bad exemplars, and how to decide between this and hand-written few-shot.

    
    
      
  1. 
        
    1
    The core analogical prompt — recall examples, then solve
    
        
    The whole technique is one appended instruction: before answering, have the model write out a few relevant solved problems of its own, then solve the target by analogy to them. No example bank required — the model curates the exemplars from its own pretraining.
    
        
    const ANALOGICAL = `Problem: ${problem}

# Instructions
Before solving, recall THREE distinct, relevant problems you already
know how to solve that are similar to this one. For each, state the
problem and its solution. THEN, using those as analogies, solve the
original problem.

# Relevant problems:
(you generate them here)

# Solution to the original problem:`;

const reply = await llm(ANALOGICAL);
// the model writes its own exemplars, then reasons by analogy

  2. 
        
    2
    Give every self-generated exemplar its own reasoning (analogical + CoT)
    
        
    The exemplars are only useful if they carry a worked method, not just an answer. Ask the model to solve each recalled problem step by step — the reasoning pattern is what transfers to the target. This is analogical prompting done as the paper intends: self-generated exemplars, each with a chain of thought.
    
        
    const ANALOGICAL_COT = `Problem: ${problem}

Recall 3 relevant solved problems. For EACH, show the full
step-by-step working, not just the final answer.

Then solve the original problem, reusing the method from the
examples and showing your steps.`;

// each exemplar demonstrates the METHOD, which is what transfers
    
      
    3. 
      
  3. 
        
    3
    Control how many exemplars — and keep them diverse
    
        
    Three to five is the sweet spot. Too few and the signal is weak; too many and the model drifts, pads latency, or starts repeating near-identical examples. Ask explicitly for a fixed count of distinct problems so you cover more of the method space instead of the same trick three times.
    
        
    const K = 3;   // 3-5 works best; tune per task
const prompt = `Recall ${K} relevant example problems that are
DISTINCT from one another (different numbers, framings, or sub-methods
— not three copies of the same problem). Solve each, then solve:
${problem}`;

// diverse exemplars = broader method coverage, less over-anchoring

  4. 
        
    4
    The cousin — Generated Knowledge Prompting
    
        
    Same family, different scaffold: instead of recalling solved problems, have the model first generate the relevant facts / knowledge, then answer using them. Use it when the failure is missing background knowledge rather than a missing method. Both share one idea — make the model produce its own scaffolding before it commits to an answer.
    
        
    // 1) generate knowledge
const facts = await llm(
  `List key facts and principles relevant to: ${question}`);

// 2) answer using the generated knowledge
const answer = await llm(
  `Knowledge:\n${facts}\n\nUsing the above, answer: ${question}`);

// analogical -> recall EXAMPLES; generated-knowledge -> recall FACTS

  5. 
        
    5
    Guard against bad self-generated exemplars
    
        
    Self-generated examples can be wrong, irrelevant, or hallucinated — and a bad exemplar actively misleads the solution that leans on it (garbage in, garbage out). Ask the model to check each exemplar's relevance and its own answer before reusing it, and keep a direct-solve fallback for when the recalled examples don't actually match.
    
        
    const prompt = `Recall 3 relevant solved problems. For each, VERIFY
its answer is correct and state how it is relevant to the target.
Discard any that are wrong or only superficially similar.
Then solve using only the examples that survived:
${problem}`;

// wrong/irrelevant exemplar -> misleads the analogy -> fall back
const answer = usableExamples > 0 ? viaAnalogy : await llm(problem);

  6. 
        
    6
    Analogical vs hand-written few-shot — when to pick which
    
        
    Reach for analogical when you have no good examples on hand, when the problem space is wide and varied (one fixed example set can't cover it), or when you don't want to maintain an example bank — the model tailors fresh exemplars to each input. Stick with hand-written few-shot when you need a specific output format or have vetted gold examples whose correctness you can't leave to the model.
    
        
    function chooseStrategy(task) {
  if (task.hasVettedExamples || task.strictFormat)
    return "few-shot";        // you control the exemplars
  if (task.wideVariety || task.noExampleBank)
    return "analogical";      // model curates per-input exemplars
  return "analogical";        // safe default on novel problems
}
