🎛️ Orchestrator–Workers — instead of hardcoding the subtasks up front, a lead "orchestrator" model reads the task and decides the plan at runtime — how many subtasks and what each one is — dispatches them to worker models, then a synthesizer stitches the results into one answer. It's the dynamic successor to parallelization's fixed split: the number of workers varies with the task.

Don't hardcode the subtasks. Let a lead model read the task and decide the plan — how many workers, and what each does — then synthesize.

Parallelization (Day 37) fans work out to a fixed set of subtasks you wrote in advance — headline, features, FAQ, tweet, always those four. But you don't always know the pieces until you see the input: a bug fix wants diagnose → test → patch, a comparison wants one worker per competitor you decide is relevant, a trip wants a track per day. Orchestrator–Workers makes the decomposition itself a model call. A lead orchestrator reads the task and emits a plan at runtime — a variable number of subtasks, each with a role and a goal — then those subtasks are dispatched to workers (the same parallel fan-out as Day 37), and finally a synthesizer reconciles the pieces into one coherent result. The count is no longer a constant in your code; it's whatever the task actually needs. Below: pick a task and watch the orchestrator produce a different-sized plan for each one — then flip to the 🔒 fixed split to see how a rigid 4-slot template mis-fits the very same task. Everything is scripted/deterministic client-side (no live model), but faithful to how a real orchestrator loop runs.

🔑 Orchestrator–Workers = parallelization where the plan is a model call, not a constant. A lead orchestrator reads the task and decides the decomposition at runtime — the number of subtasks and what each is — so it can dispatch three workers to a bug fix, five to a trip plan, or two to a tiny refactor. The workers fan out exactly like Day 37, but the list came from the model, not from your source. A synthesizer then does more than concatenate: it reconciles overlaps and smooths the pieces into one voice. You pay for the flexibility — two extra calls (the plan and the synthesis) on top of the N workers, plus the risk the orchestrator over- or under-decomposes — so reach for it when you can't know the subtasks up front. When you can, a fixed fan-out is cheaper.

🔨 Build it yourself

    
Orchestrator–Workers is three moves: plan → dispatch → synthesize. The only new idea over Day 37 is that the subtask list comes from a model call, so it's dynamic. Here is the whole thing: the fixed fan-out you're upgrading, the orchestrator, plan validation, the worker dispatch, the synthesizer, and the assembled loop.

    
    
      
  1. 
        
    1
    The fixed fan-out you're upgrading (Day 37)
    
        
    First, what almost works: parallelization with a hardcoded subtask list. You wrote the four slots before you ever saw the input, and every task gets the same four — great, until the task needs two subtasks, or seven, or entirely different ones. The decomposition is frozen in your source.
    
        
    // ❌ Fixed: you hardcode the subtasks BEFORE seeing the task.
const SUBTASKS = ["headline", "features", "faq", "tweet"]; // always these 4
const parts = await Promise.all(
  SUBTASKS.map(s => llm(promptFor(s) + input))
);
return parts.join("\n\n");
// fine — until the task needs 2 subtasks, or 7, or different ones.

    2
    The orchestrator — plan the subtasks at runtime
    
        
    The one new idea. Instead of a constant list, ask a lead model to produce the plan from the task, as structured JSON. The model decides how many subtasks and which — a role and a goal for each. This is the whole difference: decomposition becomes a call, so it adapts to the input.
    
        
    // ✅ Ask a lead model to PLAN the subtasks at runtime, as JSON.
const plan = await llm(`
  You are an orchestrator. Break this task into the RIGHT
  number of independent subtasks — no more, no fewer.
  Return JSON: { "subtasks": [ { "role": "...", "goal": "..." } ] }
  TASK: ${task}
`);
// the model decides HOW MANY and WHICH — not you, not up front.
    
      
    3. 
      
  3. 
        
    3
    Parse & validate the plan — N is now dynamic
    
        
    A model-authored plan is data you don't control, so guard it before you act on it. Parse the JSON, reject an empty plan, and cap the worker count so a runaway orchestrator can't fan out fifty calls. After this, N is whatever the task genuinely needed.
    
        
    // ✅ Parse the plan and guard it — models can return junk.
const { subtasks } = JSON.parse(plan);
if (!Array.isArray(subtasks) || subtasks.length === 0)
  throw new Error("orchestrator returned no plan");
const capped = subtasks.slice(0, MAX_WORKERS);  // don't fan out 50
// N = capped.length — dynamic, sized to THIS task.
    
      
    4. 
      
  4. 
        
    4
    Dispatch the workers — parallel fan-out (like Day 37)
    
        
    Now it's just parallelization: fan the planned subtasks out to workers with Promise.all, so they run at once and the wall clock is the slowest one. Each worker is a focused specialist — it gets its role, its goal, and the shared context. The mechanic is identical to Day 37; the only change is that the list came from the model.
    
        
    // ✅ Fan the PLANNED subtasks out to workers (parallel).
const results = await Promise.all(
  capped.map(st =>
    llm(`Role: ${st.role}. Goal: ${st.goal}. Context: ${task}`))
);
// same fan-out as Day 37 — but the subtask list was decided at runtime.
    
      
    5. 
      
  5. 
        
    5
    The synthesizer — reconcile, don't just concatenate
    
        
    Fixed sectioning could get away with join() because you designed the slots to fit together. A dynamic plan can't — the workers didn't know about each other, so their outputs overlap, contradict, or jar. The synthesizer is a final model pass that stitches them into one coherent whole: resolve overlaps, fix transitions, keep a single voice.
    
        
    // ✅ Synthesize — a FINAL model pass that reconciles the pieces.
const final = await llm(`
  Combine these worker outputs into one coherent result.
  Resolve overlaps, fix transitions, keep one voice.
  ${results.map((r, i) => `#${i + 1}: ${r}`).join("\n")}
`);
return final; // more than concat — it stitches N pieces into a whole.

  6. 
        
    6
    Put it together — and know the limits
    
        
    Three moves: plan → dispatch → synthesize. Reach for it when you can't know the subtasks in advance — coding agents, deep research, multi-file edits. Mind the limits: it costs N + 2 calls (the plan and the synthesis are the price of dynamism); the orchestrator can over- or under-decompose, so cap N and keep the plan reviewable; and the workers can't share results, so truly dependent steps still belong in a chain. Next you'll close the loop with an evaluator that grades the output and sends it back for another pass.
    
        
    // ✅ Orchestrator–Workers = plan -> dispatch -> synthesize.
async function orchestrate(task) {
  const plan   = await planSubtasks(task);      // 1 lead call, dynamic N
  const pieces = await runWorkers(plan, task);  // N worker calls, parallel
  return await synthesize(pieces, task);        // 1 synthesis call
}
// cost = N + 2 calls; N varies per task; cap N vs rate limits.
