PromptFromZero · Day 33

🔮 Automatic Prompt Engineer (APE) — stop hand-writing the instruction: define the task with a few input→output examples, let a model generate many candidate instructions, score each on how well it reproduces held-out examples, and select the winner — then resample near it for an even better one (Zhou et al., 2022)

You have a handful of examples but no idea what to tell the model. Let the model write the instruction for you — many drafts, scored on held-out examples, best one wins.

Hand-writing a good prompt is guesswork: you try a phrasing, eyeball a few outputs, tweak, repeat. Automatic Prompt Engineer (Zhou et al., 2022) turns that guesswork into search. Give it a task defined only by a few input→output demonstrations. A model then (1) proposes several candidate instruction strings inferred from the demos — "Write the antonym of the input word", "Give the opposite", "Reverse the meaning"… ; (2) each candidate is scored by how many held-out examples it reproduces correctly (execution accuracy); (3) the top-scoring instruction is selected; and (4) an optional round 2 resamples variations near the winner, which usually scores higher still. Pick a task, step through the two rounds, and watch the candidates sort by score, the winner light up, and the best accuracy climb. Everything is scripted and deterministic — no live model — but faithful to APE's propose → score → select loop.

🔑 The prompt was never written by a human — it was searched for. APE treats the instruction as the thing to optimize and the model as both the proposer (it drafts candidate instructions from the demos) and the executor (it runs each candidate to see how it scores). Because the scoring is objective — execution accuracy on a held-out set the proposer never saw — you can rank many phrasings automatically and keep the best, then resample near it to climb higher. It routinely finds instructions a human would not think to write, and it costs many model calls (proposals × held-out evaluations). Its two hard requirements: a dev set of examples and a scorer that can grade an output. Where those exist, APE — and its descendants like DSPy's optimizers — turns prompt engineering into gradient-free program search.

Propose candidate instructions from demos → score each on a held-out set → select the argmax → resample near the winner. More candidates and a bigger dev set cost more evaluations but search more of the instruction space.

🔨 Build it yourself

    
APE is a tiny optimization loop around one model: describe the task with a few examples, ask the model to propose instructions, score each proposal against a held-out set, keep the best, and optionally resample near the winner. No fine-tuning, no gradients — just generate, evaluate, select. Here is the whole thing.

    
    
      
  1. 
        
    1
    Define the task as demos + a held-out scoring set
    
        
    You do not write the instruction — you provide examples. Split them: a few demos the proposer will read to infer the instruction, and a held-out set it never sees, reserved for scoring. The held-out split is what keeps a candidate honest.
    
        
    // 1. The task is just input->output pairs. Split them.
const demos = [
  { in: "happy", out: "sad" }, { in: "hot", out: "cold" },
  { in: "fast",  out: "slow" }, { in: "up",  out: "down" },
];
const heldout = [                 // proposer never sees these
  { in: "light", out: "dark" }, { in: "open", out: "closed" },
  { in: "wet",   out: "dry"  }, { in: "loud", out: "quiet" },
];
    
      
    2. 
      
  2. 
        
    2
    Generate candidate instructions (forward generation)
    
        
    Show the model the demos and ask it to infer the instruction that maps input to output — sampled several times (nonzero temperature) so you get a diverse pool of phrasings to search over.
    
        
    // 2. Ask the model to WRITE the instruction, N times.
function proposePrompt(demos) {
  const shown = demos.map(d => `Input: ${d.in}\nOutput: ${d.out}`).join("\n\n");
  return `Here are input-output examples:\n\n${shown}\n\n` +
    `Write the instruction that maps each input to its output.`;
}
const candidates = await Promise.all(
  Array.from({ length: 8 }, () =>
    llm(proposePrompt(demos), { temperature: 0.9 }))   // diverse drafts
);
    
      
    3. 
      
  3. 
        
    3
    Score each candidate on the held-out set
    
        
    This is the objective function. Run each candidate instruction on every held-out input and count how many outputs match — execution accuracy. (Log-probability of the correct output is the other classic scorer.) No human judgement in the loop.
    
        
    // 3. Execution accuracy: does the instruction reproduce held-out?
async function score(instruction, heldout) {
  let correct = 0;
  for (const ex of heldout) {
    const pred = await llm(`${instruction}\n\nInput: ${ex.in}\nOutput:`);
    if (pred.trim().toLowerCase() === ex.out) correct++;
  }
  return correct / heldout.length;          // 0..1
}
const scored = await Promise.all(
  candidates.map(async c => ({ instruction: c, acc: await score(c, heldout) }))
);
    
      
    4. 
      
  4. 
        
    4
    Select the best (argmax)
    
        
    Sort by score and take the top instruction. That is your automatically-engineered prompt — chosen by measured accuracy, not by how good it sounded to a human.
    
        
    // 4. Keep the highest-scoring instruction.
scored.sort((a, b) => b.acc - a.acc);
const best = scored[0];
// best.instruction -> "Write the antonym of the input word."  (acc 0.83)
    
      
    5. 
      
  5. 
        
    5
    Resample around the winner (iterative Monte-Carlo)
    
        
    Optional but powerful: ask the model for variations of the current best instruction, score those too, and keep the new champion if it beats the old. This local search around high-scorers climbs toward phrasings a single proposal round would miss.
    
        
    // 5. Generate near-neighbours of the winner, re-score, keep the best.
async function resample(best, heldout) {
  const variants = await llm(
    `Generate 6 varied rephrasings of this instruction, same meaning:\n"${best.instruction}"`
  );                                                  // e.g. add "single word", "exact opposite"
  const rescored = await Promise.all(
    variants.map(async v => ({ instruction: v, acc: await score(v, heldout) }))
  );
  return [best, ...rescored].sort((a, b) => b.acc - a.acc)[0];   // >= old best
}
const winner = await resample(best, heldout);         // acc 0.83 -> 1.00
    
      
    6. 
      
  6. 
        
    6
    Mind the cost — and know when it applies
    
        
    The bill is candidates × held-out examples model calls per round, times the rounds — easily hundreds of calls to tune one prompt. That is fine because you pay it once, offline, then ship the winning instruction. But APE only works when you have the two things it needs: a labelled dev set and a scorer. No examples, or no way to grade an output, means no APE — reach for a hand-written prompt or an LLM-judge instead.
    
        
    // 6. Count the cost; require a dev set + a scorer.
const calls = candidates.length * heldout.length * rounds;   // offline, one-time
if (!haveDevSet || !haveScorer) useHandWrittenPrompt();      // APE needs both
else ship(winner.instruction);      // then reuse the found prompt in prod
// cousins: DSPy optimizers, OPRO, evolutionary prompt search -- same idea.

You learned Automatic Prompt Engineer (APE, Zhou et al., 2022): describe a task with a few input→output demos, let a model propose candidate instructions, score each by execution accuracy on a held-out set, select the argmax, and optionally resample near the winner to climb higher. The key shift is treating the prompt as something to search for rather than hand-write — the model is both the proposer and the executor, and an objective scorer replaces human eyeballing. It finds phrasings people miss, but it needs a labelled dev set and a scorer, and it spends many calls (candidates × evaluations) — paid once, offline, then the winning instruction is reused in production. It is the ancestor of today's gradient-free prompt optimizers (DSPy, OPRO, evolutionary search). 👉 Tomorrow — Day 34: Chain-of-Density — squeeze a summary tighter and tighter, packing in more entities per word with each pass.
