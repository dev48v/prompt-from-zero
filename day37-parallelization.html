⚡ Prompt Parallelization — instead of one prompt grinding through everything in a single slow serial pass, fan the same input out to several prompts that run at once, then aggregate their results: split a task into independent subtasks (sectioning) or run the same prompt N times and take the majority (voting). You win latency, or reliability — at the cost of N× the tokens.

Don't grind through everything in one long serial pass. Fan the work out to N prompts at once — then aggregate.

When a task naturally splits into independent pieces, running them one after another wastes wall-clock time, and asking one model once makes you hostage to a single flaky sample. Parallelization fixes both by fanning the same input out to several prompts that run at the same time, then combining their outputs with an aggregator. It comes in two flavors. Sectioning splits a task into independent subtasks (write the headline, the features, the FAQ, the tweet), runs each in its own focused prompt in parallel, and merges the pieces — the wall clock drops from the sum of the calls to the slowest single one. Voting runs the same prompt N times and aggregates by majority — random noise cancels out, so a shaky one-shot answer becomes a confident, self-consistent one. Both cost you N× the tokens. Below: pick a flavor and a task, choose sequential vs parallel, and watch the calls fire — see the wall-clock win, the majority-vote reliability win, and the token cost. Everything is scripted/deterministic client-side (no live model), but faithful to how a real fan-out runs.

🔑 Parallelization is map/reduce for prompts: fan one input out to N calls that run at once, then reduce them with an aggregator. Sectioning buys you latency — independent subtasks that used to run back-to-back now finish in the time of the slowest one, and each runs in its own focused prompt so the pieces are sharper too. Voting buys you reliability — running the same prompt N times and taking the majority cancels the random noise that makes a single sample flaky, and the vote share gives you a confidence number for free. The bill is the same for both: N calls means roughly N× the tokens, so reach for parallelization when the pieces are genuinely independent (sectioning) or when a correct answer is worth several tries (voting) — not to speed up steps that must feed each other in order (that's chaining, Day 35).

Prompt parallelization = fan the same input out to several prompts that run concurrently, then aggregate — sectioning (split into independent subtasks → merge) or voting (same prompt ×N → majority + confidence). Win latency or reliability; pay N× the tokens.

🔨 Build it yourself

    
Parallelization is just fan-out → aggregate — a Promise.all and a reduce. No framework required. Here is the whole thing: the slow serial version, sectioning, the parallel fan-out, the aggregator, and voting.

    
    
      
  1. 
        
    1
    The slow serial version you're replacing
    
        
    First, the thing that almost works: run each piece one after another with await in a loop. It's correct, but every call blocks the next — even though none of them actually depend on each other — so the wall clock is the sum of all the calls. (Cramming them into one giant prompt is the other trap: it's a single call, but thin and rushed at every piece.)
    
        
    // ❌ Serial: each independent call blocks the next -> wall clock = SUM.
async function serial(subtasks, input) {
  const parts = [];
  for (const t of subtasks) {
    parts.push(await llm(t.prompt + input));   // waits for each, in turn
  }
  return merge(parts);
}
// 4 calls of ~1.5s each -> ~6s, even though none depend on another.

    Sectioning — split into independent subtasks
    
        
    The first flavor. Break the task into pieces that don't need each other's output, and give each its own small, focused prompt. A launch brief becomes: headline, features, FAQ, tweet — four narrow prompts instead of one bloated one. Each is sharper on its own job, and because they're independent, they're ready to run at the same time.
    
        
    // ✅ Sectioning: one input -> N independent, focused subtask prompts.
const SUBTASKS = [
  { key:"headline", prompt:"Write a punchy headline + tagline for: " },
  { key:"features", prompt:"List the 3 flagship features of: " },
  { key:"faq",      prompt:"Write 2 FAQ Q&As for: " },
  { key:"tweet",    prompt:"Draft a 1-line launch tweet for: " },
];
// each carries ONLY its own job -> smaller, sharper than one mega-prompt.

    Fan out with Promise.all — fire them all at once
    
        
    This is the whole trick. Instead of awaiting in a loop, kick off every call without awaiting, collect the promises, and await Promise.all once. Now they run concurrently, so the wall clock is the slowest single call — not the sum. Four 1.5s calls finish in ~1.5s, not ~6s.
    
        
    // ✅ Parallel fan-out: start them all, then await once. Clock = MAX, not SUM.
async function fanOut(subtasks, input) {
  const calls = subtasks.map(t => llm(t.prompt + input)); // start ALL now
  return await Promise.all(calls);                        // wait for the slowest
}
// 4×1.5s serially = ~6s;  fanned out = ~1.5s. Same tokens, 4× less waiting.

    The aggregator — merge, or take the majority
    
        
    Fan-out is only half of it; the results have to come back together. The aggregator is the reduce. For sectioning it's a merge — stitch the sections into one document (sometimes with a final synthesis pass). For voting it's a tally — count the answers and take the majority, and the winning share is your confidence.
    
        
    // ✅ Aggregate = the reduce. Two shapes:
const merge = (parts) => parts.join("\n\n");        // sectioning: stitch pieces

function majority(votes) {                          // voting: tally + confidence
  const c = {}; votes.forEach(v => c[v] = (c[v]||0) + 1);
  const winner = Object.keys(c).sort((a,b)=>c[b]-c[a])[0];
  return { answer: winner, confidence: c[winner] / votes.length };
}

    Voting — same prompt ×N for reliability
    
        
    The second flavor uses the same fan-out, but every call is the identical prompt (run warm, so samples vary). A single sample can be a coin-flip; run it N times and the majority cancels the random noise — a shaky one-shot becomes a confident, self-consistent answer, with the vote share as a built-in confidence score. Great for classification, extraction, and tricky reasoning where correctness matters more than cost.
    
        
    // ✅ Voting: fan out the SAME prompt N times, then take the majority.
async function vote(prompt, n = 5) {
  const runs = Array.from({ length: n }, () => llm(prompt)); // identical prompt
  const votes = await Promise.all(runs);                     // all at once
  return majority(votes.map(parse));   // { answer, confidence } — noise cancels
}
// 2 of 5 samples fell for the trap answer; the majority still lands on truth.

    Put it together — and know the limits
    
        
    Both flavors are the same two moves: fanOut → aggregate. Reach for it when the work is genuinely independent (sectioning) or when a right answer is worth several samples (voting). Mind the limits: you pay N× the tokens; steps that feed each other can't be parallelized (that's chaining); and provider rate limits can throttle a big fan-out, so cap concurrency. Parallelization is a core agent building-block — next you'll add a lead model that dynamically decides the subtasks and hands them to workers.
    
        
    // ✅ Both flavors, one shape: fan out -> aggregate.
async function parallelize(input, mode) {
  if (mode === "section")
    return merge(await fanOut(SUBTASKS, input));   // latency win
  if (mode === "vote")
    return vote(input, 5);                         // reliability win
}
// only for INDEPENDENT work; costs N× tokens; cap concurrency vs rate limits.
