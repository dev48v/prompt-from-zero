PromptFromZero · Day 35

🔗 Prompt Chaining — instead of one mega-prompt that must extract, reason and write all at once, split the task into a pipeline of small focused prompts where each step's output becomes the next step's input (extract → transform → summarize); every stage does exactly one job, so accuracy climbs and you can inspect exactly where it broke

Don't ask one prompt to do everything. Break the task into a chain — and step through it to watch each stage's output flow into the next.

A single mega-prompt that must extract facts, reason over them, and write the final answer in one shot is doing three jobs at once — so it drops details, mixes up its priorities, and hands you one opaque paragraph you can't debug. Prompt Chaining decomposes that into a pipeline: a sequence of small, single-purpose prompts where each step's output is the next step's input. A classic shape is extract → transform → summarize. Each step is easy for the model because it only does one thing; each intermediate result is a checkpoint you can read, validate, and fix in isolation. Below: pick a task, then flip between the single mega-prompt and the 3-step chain. Step through the chain one prompt at a time and watch the state pass from stage to stage — everything is scripted and deterministic (no live model), but faithful to how a real chain runs.

🔑 A chain trades one hard call for three easy ones — and buys you visibility. Each step in the pipeline has a single, narrow job, so the model performs it far more reliably than when it has to juggle extraction, reasoning and writing in one breath. Just as important: every intermediate result is a checkpoint. When the answer is wrong you don't stare at one opaque paragraph — you open the Extract output, the Transform output, the Summarize output, and see exactly which stage failed. You can validate between steps, retry a single stage, or swap one prompt without touching the rest. The cost is more calls and more latency, so you chain when the task genuinely has distinct phases — not to over-engineer a one-liner.

Prompt chaining = decompose a task into a pipeline of single-purpose prompts, pass each step's output as the next step's input, and gate/inspect between steps. More reliable and far more debuggable than a single mega-prompt — at the price of extra calls.

🔨 Build it yourself

    
A prompt chain is just a small pipeline: one function per step, and the output of each step handed to the next. No framework required. Here is the whole thing — an extract → transform → summarize chain with validation gates between the stages, and the mega-prompt it replaces for comparison.

    
    
      
  1. 
        
    1
    The mega-prompt you're replacing
    
        
    First, the thing that almost works: one prompt that has to do everything. It's tempting because it's one call — but the model has to extract, reason and write simultaneously, and you get a single opaque blob back. When it's wrong, you have no idea which part broke.
    
        
    // ❌ One prompt, three jobs at once. Fluent, but unreliable + opaque.
async function megaPrompt(rawReviews) {
  return llm(`Read every review, find the single most common complaint,
             quantify it, and write a one-paragraph exec summary with a
             recommendation.\n\nReviews:\n${rawReviews}`);
}
// returns ONE paragraph. No checkpoints. Can't tell where it slipped.
    
      
    2. 
      
  2. 
        
    2
    Step 1 — Extract (narrow, structured)
    
        
    The first link does only extraction: turn messy input into clean, structured rows. Ask for a machine-readable shape (one line per item, or JSON) — no analysis, no prose. A tiny, single-purpose prompt is something the model gets right almost every time.
    
        
    // ✅ Step 1: extract only. Structured out, nothing else.
async function extract(rawReviews) {
  const out = await llm(`For EACH review, emit one line:
    + positives  ·  - negative theme in {app,battery,shipping,price,comfort,sound}
    Reviews:\n${rawReviews}`);
  return out;                       // e.g. "R1 +overall · -app ..."
}
    
      
    3. 
      
  3. 
        
    3
    Step 2 — Transform (operate on the structure)
    
        
    The second link takes step 1's output as its input and does the reasoning: aggregate, count, rank, classify. Because it works on clean rows rather than raw text, the reasoning is far more reliable — the hard parsing already happened upstream.
    
        
    // ✅ Step 2: transform. Its INPUT is step 1's OUTPUT.
async function transform(rows) {
  return llm(`From these tagged rows, tally NEGATIVE mentions per theme,
             sort descending, and list POSITIVE themes separately.
             Rows:\n${rows}`);      // rows === extract()'s return value
}
    
      
    4. 
      
  4. 
        
    4
    Step 3 — Summarize (present, don't compute)
    
        
    The last link only writes prose, and only from the tally step 2 produced. Separating "figure out the answer" from "phrase the answer" means the writing step can't quietly change the facts — it just presents what the pipeline already computed.
    
        
    // ✅ Step 3: summarize. Presentation only, over step 2's tally.
async function summarize(tally) {
  return llm(`Using ONLY this tally, write one paragraph:
             overall sentiment, the #1 complaint WITH its count,
             and a concrete recommendation.\n\nTally:\n${tally}`);
}
    
      
    5. 
      
  5. 
        
    5
    Chain them — with validation gates between steps
    
        
    The pipeline is just the three calls in order, each fed the previous result. The bonus of visible checkpoints: you can gate between steps — validate the shape, and fail (or retry just that one step) the moment something looks wrong, instead of discovering it in the final answer.
    
        
    // ✅ The chain. State flows down; gates catch problems early.
async function run(rawReviews) {
  const rows  = await extract(rawReviews);
  assert(rows.includes("-"), "extract produced no tagged rows");   // gate

  const tally = await transform(rows);
  assert(/\d/.test(tally), "transform produced no counts");        // gate

  const answer = await summarize(tally);
  return { rows, tally, answer };   // keep intermediates → debuggable!
}
// if the answer is wrong, open rows/tally to see WHICH step failed.
    
      
    6. 
      
  6. 
        
    6
    Know when to chain — and when not to
    
        
    Chaining shines when a task has distinct phases (parse, then reason, then write), when reliability and debuggability matter, or when you want validation between stages. The cost is real: more calls, more latency, and an error in an early step propagates downstream. For a simple, single-shot task, one good prompt is better — don't build a pipeline for a one-liner.
    
        
    // Decide before you build:
if (taskHasDistinctPhases && needReliabilityOrDebugging) {
  usePromptChain();          // extract -> transform -> summarize
} else if (isSimpleOneShot) {
  useSinglePrompt();         // a chain would just add latency
}
// watch out: early-step errors flow downstream -> gate aggressively.
